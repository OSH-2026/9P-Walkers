## Problem Statement

构建一个 Agent 运行沙箱，为 AI Agent 提供安全的隔离执行环境。Agent 在执行不可信指令时，可能造成：
- 文件系统破坏（误删/篡改系统文件）
- 资源耗尽（无限循环、内存溢出、fork 炸弹）
- 网络滥用（发送垃圾邮件、扫描内网）
- 权限提升（尝试获取 root 权限）

沙箱需要拦截这些危险操作，同时让正常操作顺利执行。

## What Makes This Cool

如果你能让 AI Agent 在沙箱中安全执行指令，那意味着：
1. **任何指令都可以放心执行** — 不再担心"rm -rf /"的灾难
2. **资源可控** — Agent 永远不会把你的机器跑死机
3. **可观测** — 清楚看到 Agent 在沙箱里做了什么

这对于构建可靠的 AI Agent 系统至关重要。

## Constraints

- **课程要求**：C++23 + CMake
- **时间限制**：需在大作业截止日期前完成
- **技术储备**：需学习 Linux 安全机制（namespace、cgroups、seccomp）
- **目标**：可运行的 Demo，能展示隔离效果

## Premises

1. Agent 沙箱需要在内核层（namespace/cgroups）和用户层（策略管理）配合工作
2. 使用 Linux native 的隔离机制比纯用户态模拟更高效、更安全
3. C++23 可以完成此项目（语言待定）
4. 核心功能（文件系统隔离 + 资源限制）足够完成一个可演示的 Demo

## Approaches Considered

### Approach A: 最小可行方案（C++ + Linux Native）

**Summary**: 使用 C++ + Linux namespace/cgroups 实现容器级别隔离，专注核心功能。

**Effort**: S/M（约 2-3 周）
**Risk**: Low
**Pros**:
- 利用现有 C++ 项目结构
- 代码量适中，易于理解和维护
- 隔离效果可靠（Docker 验证过的技术）

**Cons**:
- 功能相对基础
- 需要学习 Linux API

**Reuses**: lab0 的 CMake 构建结构

---

### Approach B: 混合增强方案（Go + eBPF）

**Summary**: Go 编写用户态沙箱核心 + eBPF 监控层，更现代的技术栈。

**Effort**: M/L（约 3-4 周）
**Risk**: Med
**Pros**:
- eBPF 是当前热点技术，有展示价值
- 可视化监控 Agent 行为
- 技术栈更现代

**Cons**:
- Go 语言需要额外学习
- eBPF 调试复杂
- 依赖内核版本（需要较新内核）

**Reuses**: 较少，需从头搭建

---

### Approach C: 全栈完整方案（C 内核模块）

**Summary**: C 语言编写内核模块 + 用户态管理界面，真正硬核 OS 项目。

**Effort**: L/XL（约 4-6 周）
**Risk**: High
**Pros**:
- 真正的内核级开发经验
- 技术深度最高
- 对理解 OS 有极大帮助

**Cons**:
- 学习曲线非常陡峭
- 调试困难（kernel panic 风险）
- 难以在 Demo 环境中演示

**Reuses**: 较少

---

## Recommended Approach

**选择 Approach A（最小可行方案）** 原因：
1. 时间有限，需在截止日前完成
2. 需同时学习 OS 概念，不宜再增加语言学习负担
3. namespace/cgroups 是 Docker/Kubernetes 的基石，工业界验证充分
4. 功能完整后可扩展至 Approach B 的监控层

**Completeness: 8/10** — 覆盖核心隔离场景，但未涉及网络隔离和高级安全策略。

2. 资源限制生效（限制内存、CPU、进程数）

4. 可通过日志/trace 查看 Agent 执行行为
5. Demo 成功演示隔离效果
