# 4.20-4.21 mini9P协议的实现

**Auther: 武文韬**

当前 mini9P 实现属于“能跑通最小链路的客户端原型”，对课程展示的基础目标是基本正常的，但距离“稳定可演示、可扩展”的 v1 完整落地还有几块关键短板。

**现状判断**

协议编解码层整体是通的，帧格式、CRC、消息类型、错误码都实现了，基础质量不错。
客户端 7 个核心操作都已实现，可支撑低并发下的 ls/cat/echo/stat 基本交互。
目前更像“单链路同步 RPC 客户端”，不是完整 mini9P 子系统：可靠性机制和状态机约束还不够。

**目前已满足的功能**

1. 帧收发与校验
   - 编码/解码与 CRC16 已实现，见 mini9p_protocol.c:89 和 mini9p_protocol.c:98
   - CRC 算法符合规范，见 mini9p_protocol.c:61
2. v1 核心消息对
   - Tattach/Twalk/Topen/Tread/Twrite/Tstat/Tclunk 的构建函数都在 mini9p_protocol.h:122
   - Rattach/Rwalk/Ropen/Rread/Rwrite/Rstat/Rerror 的解析函数都在 mini9p_protocol.h:172
3. 客户端调用链
   - attach/walk/open/read/write/stat/clunk 全部已实现，见 mini9p_client.c:84
   - 还有 walk_path/open_path 便捷接口，见 mini9p_client.c:317
4. 目录读取辅助
   目录项解析函数已实现，见 mini9p_protocol.c:419
5. 协议对象与错误码
   Tag/Fid/Qid/feature bits/error codes 基本齐全，见 mini9p_protocol.h:24

**关键缺口与风险**

1. Twrite 被硬编码限制为 256 字节，不完全跟随 msize 协商，见 mini9p_protocol.c:250
2. 请求路径是一次性同步调用，无内建超时重传框架，见 mini9p_client.c:7
3. Fid 生命周期缺少活跃表和状态机硬约束（例如未 open 就 read/write 的防错不足）
4. 协议规范中提到的重复请求响应缓存、多 Tag 并发能力尚未形成完整机制
5. 从当前可见代码看，重点在 client+protocol；完整 server 侧能力还需要继续补齐到工程闭环

基于规范与实现对照，当前代码已具备 mini9P v1 的最小客户端能力（attach/walk/open/read/write/stat/clunk），可支撑低并发、短链路下的基础 Shell 交互；但在可靠性与协议约束方面仍有关键缺口（重传/超时、fid状态机、msize/iounit约束联动）。

目标：先把“可演示稳定版”做实，再考虑并发和 v2 扩展，避免范围失控。

**Steps**

1. Phase A - 现状结论固化（阻塞后续）：整理协议规范与代码映射，形成“已实现/未实现/风险”清单，作为团队统一基线。
2. Phase B - 协议一致性修正（依赖1）：梳理并修正与规范存在偏差的行为约束，重点是 Twrite 固定 256 限制与 msize 协商后的请求大小边界策略。
3. Phase C - 客户端可靠性增强（依赖2）：在请求发送路径加入超时与同 Tag 重试框架（保持请求体不变），并补充最小诊断日志（Tag/Type/错误码）。
4. Phase D - fid 生命周期与状态机（并行于3，建议先完成基础后再并行）：引入活跃 fid 跟踪与规则校验，覆盖“未 open 禁止 read/write”“已 open 禁止 walk”等协议约束。
5. Phase E - 集成验证与回归（依赖3、4）：围绕 ls/cat/echo/stat 与目录读取流程建立回归用例，覆盖正常路径、边界长度、CRC 错误、Tag 不匹配、超时重试、clunk 幂等。
6. Phase F - v2 预留项评估（依赖5）：评估多 Tag 并发与响应缓存的实现成本，定义是否进入下一里程碑。

**Verification**

1. 协议级自测：对每个 T/R 对进行 encode/decode 与 parse round-trip，验证字段边界（空路径、最大路径、最大 name、count=0、count=上限）。
2. 一致性检查：按规范逐条检查 attach 协商结果对后续 read/write 限制是否生效，确认不再存在硬编码上限与协商值冲突。
3. 可靠性测试：注入超时/丢帧/错 Tag/CRC 错误，验证同 Tag 重试与错误码回传行为。
4. 状态机测试：验证 fid 在 attach/walk/open/read/write/stat/clunk 生命周期中的合法与非法序列。
5. 场景回归：执行三类典型流程（读文件、读目录、写命令+轮询状态）并记录日志。

**Decisions**

- 包含范围：mini9P 协议层与客户端层的一致性和可靠性改进建议。
- 不包含范围：slave 端完整 rpc_server 设计与实现细节（仅在路线中给出接口预留方向）。
- 优先级决策：先做可演示稳定性（超时重试 + 状态机 + 边界约束），再做并发与缓存扩展。

**Further Considerations**

1. 是否在当前里程碑内引入“客户端内部 fid 表”以硬约束 max_fids（推荐是，收益高、改动中等）。
2. 是否把重传策略放在 transport 层还是 mini9p_client 层（推荐 client 层最小实现，transport 层保留物理链路重发）。
3. 是否现在就做多 Tag 并发（推荐延后到 v2，先保证单 inflight 稳定演示）。
