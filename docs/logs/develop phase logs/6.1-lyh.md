# 开发日志6.1 - lyh

## 日期

2026-06-01

## 概述

本阶段主要完成了 **PC host + MCU1(F407) + MCU2(F411)** 串联上板联调，实现了多跳 mesh 拓扑下的 bootstrap、路由学习和主机控制器下发路径。同时在 shared layer 和从机侧引入了 `NEIGHBOR_PROBE` 机制，用于解决 relay 从机不知道下游从机地址对应哪个 UART端口的问题。

## 主要工作

### 1. 上板联调（PC + MCU1 + MCU2）

完成了 `PC emulator -- mcu1 -- mcu2` 串联拓扑的首次成功联调。

**硬件连接：**

- PC USB-UART @ 1000000 baud → mcu1 USART2 (PA2/PA3)
- mcu1 USART1 (PA9/PA10) @ 1000000 → mcu2 USART2

**关键修复项：**

- STM32F411 时钟树改为 HSE PLL：SYSCLK 96MHz，APB1 48MHz，使 USART1/2 稳定运行在 1M baud
- 串口阻塞 timeout 从 200ms 改为 1ms，避免 mesh_node_service_poll_once 卡住
- 错误码域修复：`-MESH_ERR_BUSY == -6` 被误当成 `M9P_ERR_EISDIR`，导致 emulator 认为不可重试错误而退出。修复后 PC 端正确将 `MESH_ERR_BUSY` 映射为 `M9P_ERR_EAGAIN`
- 添加 `mesh_uart_transport_rx_pending()` 和 `mesh_uart_transport_flush_input()`，消除 rx 空转时的无效阻塞

### 2. NEIGHBOR_PROBE 机制

引入两个新控制帧类型，实现单链路本地邻居发现：

| 类型 | 用途 |
|------|------|
| `MESH_TYPE_NEIGHBOR_PROBE_REQUEST (0x17)` | 单链路本地请求，接收端不得转发 |
| `MESH_TYPE_NEIGHBOR_PROBE_RESPONSE (0x18)` | 单链路本地响应，用于学习 `src -> ingress_port` |

**行为约束：**

- 从机 ASSIGN完成后，向所有已启用 UART 广播 `NEIGHBOR_PROBE_REQUEST`
- 收到 `NEIGHBOR_PROBE_REQUEST` 后，若本机已有正式地址，从同一 ingress port 直接回复 `NEIGHBOR_PROBE_RESPONSE`
- 只有 `NEIGHBOR_PROBE_RESPONSE` 才建立 direct neighbor `addr -> ingress_port`映射
- 从机收到 `NEIGHBOR_PROBE_RESPONSE` 后，若已知上游 control-plane，立即向上游发送 `LINK_STATE(src=本机, neighbor=response.src)`

### 3. 主机控制器路由下发

实现了主机在收到 `LINK_STATE` 后，基于当前有向 topology 即时计算并下发 `ROUTE_UPDATE`。

**新增接口：**

```c
int cluster_compute_next_hop_from(
    const struct cluster *cluster,
    uint8_t source,
    uint8_t dst,
    uint8_t *out_next_hop,
    uint8_t *out_metric);
```

**语义：**

- 只读计算，不写入 `cluster->routes`
- 按 topology 即时计算 `source -> dst` 的 `next_hop` 和 `metric`
- 收到单向 `A -> B` 时不自动生成 `B -> A`；反向路径必须由 `B` 自行上报 `LINK_STATE`
- 一次 sync 枚举每个 `source` 到每个 `dst` 的路径，向 `source` 下发 `ROUTE_UPDATE(dst, next_hop, SET)`

### 4. 从机 relay bootstrap 支持

从机侧新增 pending register 表和 ASSIGN 回转逻辑：

**新增字段：**

```c
struct mesh_node_runtime_bootstrap_pending {
    bool used;
    uint8_t uid[MESH_UID_LEN];
    uint32_t boot_nonce;
    uint8_t ingress_port;     // 下游入口 port
    uint8_t assigned_addr;
    bool assign_forwarded;
};
```

**Relay 行为：**

- 收到下游 bootstrap `REGISTER(src=0xff, dst=0xff)` 时记录 `uid + boot_nonce -> ingress_port`，并向上游转发
- 收到命中 pending UID 的 `ASSIGN` 后，将原始 ASSIGN 从对应下游 ingress port 发出
- 学习 `child_addr -> ingress_port`，并向上游发送 `LINK_STATE(src=relay, neighbor=child)`

### 5. 从机动态 addr->port 映射

移除了从机 service 的静态 `neighbor_addr` 配置，改为运行时学习：

```c
struct mesh_node_service_addr_port {
    bool used;
    uint8_t mesh_addr;
    uint8_t port_id;
};
```

- `learn_peer_port` 回调：runtime 从 `NEIGHBOR_PROBE_RESPONSE` 学到 `src -> ingress_port` 后通知 service 记录
- `send_frame(next_hop)` 时查动态 `addr -> port` 表选择实际 UART
- `MESH_NODE_SERVICE_NEIGHBOR_ANY` 仅作为 bootstrap REGISTER 广播 selector

### 6. 调试接口 `/sys/routes`

从机新增 `/sys/routes` 虚拟文件，输出格式：

```
local=0x22 control=0x00 upstream_port=1 upstream_peer=0x11
route dst=0x11 next=0x11 metric=1 local=0
route dst=0x00 next=0x11 metric=1 local=0
addr_port addr=0x11 port=1
```

用于验证从机路由表和 addr-port 映射是否正确。

### PC + MCU1 + MCU2 串联联调

```bash
# mcu2 先烧录
pwos-slave-stm32f411/build.sh flash

# mcu1 烧录 + 启动 emulator等待 2 节点
pwos-slave/build.sh flash-test # PC_NODE_COUNT=2
```

流程：
1. mcu1 REGISTER → PC ASSIGN → confirmed
2. mcu1 NEIGHBOR_PROBE_REQUEST → PC NEIGHBOR_PROBE_RESPONSE → mcu1 LINK_STATE
3. PC 向 mcu1 下发 ROUTE_UPDATE(dst=0x00, next=0x00)
4. mcu2 REGISTER（经 mcu1 转发）→ PC ASSIGN → mcu2 confirmed
5. mcu1 LINK_STATE(src=0x11, neighbor=0x22) // mcu1 上报下游 child
6. mcu2 LINK_STATE(src=0x22, neighbor=0x11) // mcu2 上报直连 relay
7. PC 计算拓扑，向 mcu1 下发 `ROUTE_UPDATE(dst=0x22, next=0x22)`
8. PC 计算拓扑，向 mcu2 下发 `ROUTE_UPDATE(dst=0x00, next=0x11)` 等多跳路由
9. attach mcu2 → read /mcu2/sys/health

输出：`pc_master_emulator: ok`

## 已知问题与后续工作

### 下一阶段 TODO

1. **STM32 多 UART RX 异步化**
   - 当前 `rx_pending + poll_once` 是短期止血方案
   - 后续需改用 `HAL_UARTEx_ReceiveToIdle_DMA()` 或 circular DMA + IDLE IRQ
   - 避免 RX 空转时阻塞 200ms

2. **邻居传播路由模型（备选）**
   - 当前保持"控制器下发指定路由"模型
   - 后续可评估邻居传播模型（每个节点广播可达表，接收方比较 metric）

3. **route sync 按需触发**
   - 当前每次 LINK_STATE 都触发全量 sync
   - 后续可优化为只对受影响节点下发增量 ROUTE_UPDATE
