# 开发日志 6.10 - wwt

## 日期

2026-06-10

## 概述

本次为 **ESP32-P4 主机** 补上了一直缺失的 **ASSIGN + relay bootstrap 控制器**。

此前主机固件（`mesh_host_runtime`）从不发送 ASSIGN：动态地址分配逻辑只实现在 `tools/pc_master_emulator/main.c` 里，从未移植进真正的主机运行时。后果是——从机上电后发 bootstrap `REGISTER`（src=0xFF）等待地址，但主机收到后只更新元数据、不回 ASSIGN，从机于是永远停在未分配状态。这也是「真 P4 主机连不上节点 / 多跳找不到 mcu2」的直接原因。

本次把分配控制器从 emulator 对照移植进 `mesh_host_runtime`，让 emulator 和真实主机跑**同一份**控制器逻辑，并按之前关于控制面可靠性的讨论，把**幂等**作为重传安全性的基础做进设计。

## 背景：为什么之前会卡

- 从机 `mesh_node_runtime.c:416` 只有收到匹配 UID 的 ASSIGN 才会把 `local_addr` 从 0xFF 改为正式地址；拿不到 ASSIGN 就一直未分配。
- 主机 `mesh_host_runtime_control_handler` 的 REGISTER 分支调用 `register_assigned_node(frame->src, uid)`，而该函数对 `src == UNASSIGNED` 直接 no-op；shared 的 `cluster_processor_control_handler` 的 REGISTER 分支也只更新元数据、不构造 ASSIGN。
- 全仓库 `mesh_build_assign` 的非测试调用点**只有** emulator 一处。

## 主要工作

### 1. 主机侧 ASSIGN 控制器（`mesh_host_runtime.c`）

新增两个内部函数，并接入控制帧处理：

**地址分配 `mesh_host_runtime_alloc_or_reuse_addr()`**

- 先按 UID 命中已有槽位 → **复用原地址**，保证对 REGISTER 重传幂等（不漂移、不重复挂载）。
- 新 UID 按已分配槽位数取第 n 个（1-based），地址 `(n<<4)|n`，即 `0x11、0x22 … 0xee`；`0x00`(主机)/`0xff`(未分配) 保留，最多 14 个节点。该方案与 emulator 及从机 `/sys/routes` 显示约定一致。

**bootstrap 处理 `mesh_host_runtime_handle_bootstrap_register()`**

- 分配/复用地址 → 填 `mesh_assign_payload`（uid、node_addr、lease_ms=60000、epoch=1、node_name="mcuN"）→ `mesh_build_assign(local, dst=UNASSIGNED, …)` → **经 `config.send_frame` 直接发出** → `register_assigned_node(new_addr, uid)` 绑定 mesh-backed client / VFS。
- **关键点（踩坑）**：ASSIGN 的 `dst=0xff` 没有路由，绝不能走 processer 的 `out_reply_frame` 回包路径——那会在 `mesh_processer_send_raw_frame` → `route_lookup` 处被丢弃。必须用 `send_frame` 直接发。单口主机下 `mesh_host_service_send_frame` 对任意 next_hop 都落到唯一端口；中继从机据 UID 把 ASSIGN 转发到下游（relay 转发逻辑早已在从机侧实现并验证过）。

**控制帧入口改造 `mesh_host_runtime_control_handler()`**

- 在调用 shared `cluster_processor_control_handler` **之前**拦截 bootstrap REGISTER（`type==REGISTER && src==UNASSIGNED`），避免给占位地址 `0xff` 建立无意义的 cluster 节点元数据。
- 原 REGISTER 分支收窄为只处理 confirm REGISTER（src=已分配地址），作旧固件/二次注册的幂等刷新。

### 2. 可靠性设计取舍（幂等优先）

按之前关于「注册握手 fire-and-forget、丢一帧就卡死」的讨论，本次主机侧落地的是**重传安全性的基础——幂等**：

- 从机在拿到地址前会按退避不断重发 REGISTER（从机侧重传为后续工作），主机的 bootstrap 分支天然成为 **ASSIGN 的重传驱动**，主机无需自己开 ASSIGN 定时器。
- 同一 UID 反复 REGISTER → 复发同一地址、`register_assigned_node` 幂等刷新，不产生地址漂移或重复 `/mcuN` 挂载。
- `epoch` 固定为 1，为从机侧「已是该 addr+epoch 则 no-op」留好接口。

### 3. 测试更新（`vfs_bridge/test/test_mesh_host_runtime.c`）

原 `test_unassigned_register_waits_for_assigned_registration` 断言的是**旧的错误行为**（bootstrap REGISTER 被隐藏、节点不可见），与新控制器冲突。重写为 `test_bootstrap_register_assigns_and_registers`，覆盖：

1. bootstrap REGISTER → 发出一帧 ASSIGN，dst=0xFF、addr=0x11、UID 与 REGISTER 一致；
2. 占位地址 0xFF **不被**建成 cluster 节点（验证「在 shared cluster 处理器之前拦截」）；
3. mcu1 随即可见（`cluster_vfs_attach` 返回 EAGAIN：已登记但尚未可达）；
4. 同一 UID 重复 REGISTER 幂等 → 复用 0x11；
5. 不同 UID → 分配下一个地址 0x22。

新增 helper `expect_assign_tx()` 解析 TX 队列中的 ASSIGN 帧。

## 已知问题与后续工作

本次只补齐了主机侧分配控制器与其幂等性，**控制面完整可靠性尚未做完**。按优先级：

1. **从机 REGISTER 分配前重传（P0，最关键）**：当前 `poll_neighbor_probe_retry` 在 `local_addr==UNASSIGNED` 时直接 return，分配前零重试——这是多跳卡死的主因。需让未分配时走 REGISTER 退避重发（建议 250ms ×1.5 退避、封顶 1s、分配前不限次）。本次主机幂等已为该重传铺好路。
2. **LINK_STATE 引导期重发 + ROUTE_UPDATE version 门控重推（P1）**：解决「卡在 route 分发」。
3. **多口主机 ingress-port 打通（P2）**：当前 ASSIGN 经单口广播即可；多口主机需把 ingress_port 从 `poll_once` 透传到 control_handler，按入口端口回发。
4. **同步心跳 / boot_nonce 重启处理 / 主机看门狗（P2）**。
5. **STM32 RX DMA（根因）**：当前从机仍是 `HAL_UART_Receive` 逐字节阻塞，多口并发易丢帧；重传让系统「能收敛」，DMA 让它「少丢、快收敛」，两者互补。详见 `TODO.md`。

### 建议的下一步

- 先做 P0（从机 REGISTER 重传），它与本次主机幂等配套后即可让 `host—mcu1—mcu2` 收敛；
- 再给 emulator 加丢帧注入测试，用它驱动 P1 可靠性开发。
