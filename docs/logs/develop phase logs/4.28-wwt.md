# 4.28 Web Shell 初始化与基础功能实现

**Author：武文韬**

根据 `architecture.md` 中对于 **Phase 5：Web Shell** 以及总体架构（Master 负责控制平面 Web+Shell，Slave 负责边缘计算和硬件驱动）的设计，今天开展了基于 Web 通信架构的开发任务。目标是使 ESP32-P4 （主控节点）承载一个可通过浏览器访问的全局控制台，允许用户动态通过 WiFi 观测各 STM32 从机的工作状态，甚至利用 Lua 脚本和 `mini9p` 协议完成主从操作。

## 1. Web Shell 前端架构设计 (index.html)

为了降低 ESP32 Flash 内容的占用和编译的复杂度，我们没有采用重量级的 React/Vue 打包体系，而是使用原生 **HTML + Vanilla JS + CSS** 编写了嵌入式设备的单文件前端 `pwos-master-esp32p4/main/web/index.html`。同时通过 CDN 引入 `xterm.js` 及字体图标。

前端设计的核心是**五大左右侧栏选项卡视图**：

1. 💻 **终端 (Terminal)**
   - 集成了 `xterm.js` 扩展插件，实现了完整的 Linux 规范黑框终端。
   - 所有浏览器内的键盘敲击将直接转化送入 `WebSocket`，ESP32 将基于 4.27 刚完工的 `shell_execute_line` 直接执行交互。
2. 📁 **文件管理 (Files)**
   - 一套展示 `/mcu1/`, `/mcu2/` 等远程虚拟节点的界面。
   - 通过在后台通过 `ws.send('ls /')` 或底层的 `m9p_walk` 命令读取结构返回进行列表渲染。
3. 📊 **设备监控 (Monitor)**
   - 查看运行 Master OS (ESP32-P4) 自身的 Heap 内存占用状态以及通过 `m9p_attach` 操作尝试检测并连通从机。
4. 📜 **Lua 脚本 (Scripts)**
   - 设置了支持编辑 Lua 代码段的文本区，点击“执行”时将文本自动换行压缩成单行命令：`lua print('hello')`，打入 WebSocket 令解析器在主控平台运作以调用 VFS 的 `read/write` 调度各从机计算结果。
5. ⚙️ **设置 (Settings)**
   - 核心层参数配置中心：用于配置 `pwos-slave` 与 `pwos-master` 建立的传输通道（**UART/SPI/WIFI** 三种介质）。

## 2. 主从连接方式配置与 mini9P 集成通道

在前端建立后，我们在 Web 的 **设置 (Settings)** 页面提供了三种连接到 STM32 从机的选项，这与通信层 Phase 的规划完全契合：

- **UART**（最稳定，Phase 1 & 2）：通过页面选择 115200 或 921600 波特率，向 WS 触发配置指令。ESP32 的 `mini9p_client` 的 transport 回调自动指向 uart 下发 `Tattach` 帧。
- **SPI**（高吞吐，Phase 2）：通过配置 1MHz~10MHz 的主从 SPI 频率向挂载设备发送 mini9p 序列包。
- **WiFi Mesh**（可扩展，Phase 3）：允许向设定同在一个子网下的 STM32/ESP IP 地址直发包。

> **注：以上无论哪种硬件方式，载荷内部必须严格遵从我们在 `mini9p/mini9p_protocol.c` 和 `README.md` 中实现的 9P 二进制帧格式，即以 `0x39 0x50` (9P) 开头的 10字节定长表头加 Payload 及 CRC 校验。**

## 3. 全局唯一 WebSocket 桥接方案

为了最大化 ESP32 性能并简化从机的状态管理，此次架构确立了**全局单路并发的 WebSocket 策略**：

1. **PC 端**浏览器打开 `http://192.168.4.1/` 加载出刚才设计的 `index.html`。
2. 页面中的 JS 会尝试立即握手 `ws://192.168.4.1/ws` 开启 WebSocket。
3. Master 的 HTTP C 程序通过钩子拦截原属于 UART 标准打印（printf/puts）的流，并将回显封入 WebSocket Text 帧，双向打通 PC 与系统的 `shell`。
4. Shell 根据输入的内容解作底层的 `m9p_client_walk` 或者 `lua_run_buffer` 传向目标 STM32 节点。
