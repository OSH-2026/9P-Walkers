## 一、问题诊断

在启动本次开发之前，对 `pwos-master-esp32p4` 的 Web 子系统做了完整的代码审查，发现三个根本性缺陷：

### 缺陷 1：`websocket_shell_broadcast` 是占位代码

```c
// websocket_shell.c:54（原始代码）
// TODO: 实现由 ESP-IDF HTTPD WS 发送异步帧到 Socket Client
printf("TODO_WS_BROADCAST: %s", ws_out_buffer);
```

广播函数从未向任何浏览器发送数据——仅仅往 UART 打印了一条 TODO 日志。

### 缺陷 2：Web 模块从未被编译进固件

检查 `main/CMakeLists.txt`，SRCS 列表中不包含 `http_server.c`、`websocket_shell.c`，`../web` 也不在 INCLUDE_DIRS 中，`index.html` 也没有 EMBED_FILES。整个 Web 层在编译时根本不存在。

### 缺陷 3：没有 WiFi 初始化

`app_main` 中没有任何 WiFi 相关调用，HTTP 服务器即使启动也无法被外部网络访问。

## 二、实施方案

### 2.1 Shell 输出钩子（`shell.h / shell.c`）

**问题**：shell 命令的 `printf`/`puts` 输出仅走 UART，无法被 WebSocket 层捕获。

**方案**：引入 `shell_print_hook_t` 回调机制。

```c
// shell.h 新增
typedef void (*shell_print_hook_t)(const char *text);
void shell_set_print_hook(shell_print_hook_t hook);
```

在 shell.c 内部，所有 `printf`/`puts` 调用被替换为 `shell_printf()`/`shell_puts()`，这两个函数调用 `shell_emit()`，后者同时写 stdout 并触发 hook：

```c
static void shell_emit(const char *text) {
    fputs(text, stdout);
    if (s_print_hook) s_print_hook(text);
}
```

UART 提示符 `pwos> ` 仍直接走 `fputs`，不经过 hook，避免浏览器 xterm 出现重复提示符。

**改动范围**：`shell.c` 中约 20 处 printf/puts 调用。

### 2.2 WebSocket 异步广播（`websocket_shell.c`）

**核心数据结构**：

```c
static httpd_handle_t s_server = NULL;
static int            s_client_fds[MAX_WS_CLIENTS];  // MAX = 4
static int            s_client_count = 0;
static portMUX_TYPE   s_client_lock = portMUX_INITIALIZER_UNLOCKED;
```

**发送机制**：

ESP-IDF 的 `httpd_ws_send_frame_async` 需要在 httpd task 上下文调用，且 payload 必须在回调执行时仍然有效。选用 `httpd_queue_work` + 堆分配 work item 的模式：

```c
typedef struct {
    int    fd;
    size_t len;
    uint8_t data[];  // flexible array，payload 紧跟结构体
} ws_send_work_t;

static void ws_send_work_fn(void *arg) {
    ws_send_work_t *work = arg;
    httpd_ws_frame_t frame = { .type = HTTPD_WS_TYPE_TEXT, .payload = work->data, .len = work->len };
    httpd_ws_send_frame_async(s_server, work->fd, &frame);
    free(work);  // 发完立即释放
}
```

每次广播为每个客户端独立 malloc 一个 work item，消除多客户端时的 buffer 竞争。

**命令执行流**：

```c
void websocket_shell_receive_text(const char *data, size_t len) {
    // 1. echo 提示符到浏览器
    ws_send_raw_to_all("pwos> cmd\r\n", ...);
    
    // 2. 挂钩 shell 输出
    shell_set_print_hook(shell_output_hook);
    
    // 3. 执行命令（所有 printf 自动转发到 WS）
    int rc = shell_execute_line(line);
    
    // 4. 取消钩子
    shell_set_print_hook(NULL);
}
```

### 2.3 客户端连接生命周期（`http_server.c`）

```c
// WebSocket 握手时注册客户端
if (req->method == HTTP_GET) {
    websocket_shell_client_connected(httpd_req_to_sockfd(req));
    return ESP_OK;
}

// socket 关闭时自动注销（通过 cfg.close_fn）
static void on_socket_close(httpd_handle_t hd, int sockfd) {
    websocket_shell_client_disconnected(sockfd);
}
cfg.close_fn = on_socket_close;
```

### 2.4 WiFi SoftAP（`wifi_softap.c`）

新建模块，配置 ESP32-P4 为 WPA2 接入点：

| 参数 | 值 |
|---|---|
| SSID | `9P-Walkers` |
| 密码 | `pwos1234` |
| IP | `192.168.4.1`（ESP-IDF default AP 地址） |
| 最大连接 | 4 |

IP 与 `index.html` 中的 WebSocket fallback 地址一致：
```js
const host = window.location.hostname || '192.168.4.1';
```

## 四、上板验证说明

> 本次 CI 无自动化 ESP-IDF 环境，需手动上板验证。

### 验证步骤

1. 在已配置 ESP-IDF 5.x 环境下编译 `pwos-master-esp32p4`：
   ```
   idf.py build
   idf.py flash monitor
   ```
2. 手机/PC 连接 WiFi `9P-Walkers`（密码 `pwos1234`）
3. 浏览器访问 `http://192.168.4.1`，确认 Web Shell UI 加载
4. 侧边栏状态显示绿色 `Connected`
5. 依次测试以下命令，确认浏览器 xterm 中出现对应输出：

| 命令 | 预期输出 |
|---|---|
| `help` | 命令菜单列表 |
| `heap` | `free heap: XXXX bytes` |
| `echo hello world` | `hello world` |
| `lua print(1+1)` | Lua 执行结果 `2` |
| `m9p_attach` | `mock attach result: ...` |

### 已知限制（待后续 PR 解决）

1. **并发安全**：UART shell task 和 WS 命令无互斥锁，同时触发时输出可能交织
2. **cluster_vfs 为空**：`ls`/`cat` 命令走 mock transport，无法访问真实从机
3. **无 TLS**：仅适用于本地开发，不适合生产部署
