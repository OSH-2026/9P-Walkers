# PC Master Emulator 首次上板记录

日期：2026-05-09

## 结论

Mini9P 从机侧串口闭环已经首次跑通。PC Master Emulator 能够通过 USB-TTL 与 STM32 通信，并完成以下固定流程：

1. `Tattach`
2. `Twalk /sys/health`
3. `Topen OREAD`
4. `Tread /sys/health`
5. `Tclunk`
6. 错误路径补测：`/missing` 返回 `ENOENT`
7. 非法 fid 补测：返回 `EFID`

最终 `/sys/health` 能读到 `ok\n`，工具返回成功。

## 联调命令

```bash
./pc_master_emulator /dev/ttyUSB0 1000000
```

串口参数：

- 设备：`/dev/ttyUSB0`
- 波特率：`1000000`

## 偶发异常

目前观察到两类偶发超时。

### Twalk 后超时

```text
./pc_master_emulator /dev/ttyUSB0 1000000
pc_master_emulator: /dev/ttyUSB0 @ 1000000 baud
[PC→STM32] Tattach (16 bytes)
[STM32→PC] Rattach tag=1 payload=16
   msize=512 max_fids=16 inflight=1 features=0x00000001
[PC→STM32] Twalk /sys/health (26 bytes)
serial read timeout
```

### Tattach 后超时

```text
./pc_master_emulator /dev/ttyUSB0 1000000
pc_master_emulator: /dev/ttyUSB0 @ 1000000 baud
[PC→STM32] Tattach (16 bytes)
serial read timeout
```

## 初步判断

已确认的一个问题是：加入 LED 闪烁时使用了 `HAL_Delay()`，会阻塞主循环，导致串口轮询不及时，从而引发超时。该问题后续应改为基于 `HAL_GetTick()` 的非阻塞闪烁。

另外仍出现过无法稳定复现的异常，暂时保留为待观察项。后续可重点检查（这些当然是AI写的）：

- 主循环是否仍存在阻塞路径。
- USART2 是否被其他调试输出污染。
- PC 端 timeout 设置是否过短。
- USB-TTL 连接与供电是否稳定。
- 从机复位后首帧同步是否可靠。
