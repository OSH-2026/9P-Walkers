实现落点与设计决策 07 / 08

现在已经切到 mesh transport，client 仍只依赖 transport 函数和 ctx。

端口信息不在函数名里，而在 transport_ctx 里；当前 ctx 由 mesh_host_runtime 维护，包含目标槽位、运行时状态和 cluster 派生出的下一跳信息。

当前：mesh transport

client->transport = mesh_host_runtime_client_request

可替换的 transport

m9p_client_init(&client, custom_transport, &transport_ctx)

只要满足 m9p_transport_fn 签名，就可以替换底层链路实现。

cluster manager

维护权威节点状态、能力、在线状态、端口、邻居、全局拓扑和下一跳派生结果。

cluster_vfs

迁移为 VFS 挂载表：node_name -> m9p_client、attach 状态、fd/fid 会话状态。

已决策
  • 控制面和多跳转发统一走 mesh envelope。
  • mini9P 不新增注册、命名、拓扑或路由语义。
  • 主机维护全局拓扑图；从机只维护本地邻居和下发转发表。
  • cluster/ 维护权威节点状态和能力，供调度器、诊断和链路层使用。
  • 节点 UID、节点名、节点地址三者分离，用户路径只暴露节点名。
  • cluster_vfs 迁移为 VFS 挂载表，只维护文件访问会话状态。
  • 主机侧管理沿用原架构 cluster/ 命名。
  • 旧的 mini9p_peer_link 已移除；当前活跃链路层实现是主机侧 mesh_host_runtime 与节点侧 mesh_node_runtime。
待讨论
  • bootstrap 注册的反向路径和多未分配节点去重。
  • mesh envelope 的具体字段、地址宽度、帧类型和错误处理。
  • 新 mesh/cluster 链路层命名待定：候选包括 cluster_link、cluster_mesh_link、cluster_link_transport、cluster_mesh_transport。
  • cluster/ 内部最终拆分哪些文件和接口。
  • 同步心跳的精度目标、超时策略、链路断开上报和从机转发表失效策略。

总结：用户通过节点名访问文件；mini9P 负责文件语义；mesh envelope 负责投递和控制面；cluster/ 负责主机侧节点管理与全局拓扑。

后续讨论清单 08 / 08

下一轮不要从实现细节散开，先压实会影响代码形状的问题。

每个问题都先给推荐方案，再看是否接受。优先处理 envelope、注册回程、从机转发表和 VFS 挂载边界。

1. Envelope 怎么定？

推荐：src/dst/type/seq/hop/flags/payload_len，地址先用 uint8_t。

原因：足够支撑小规模 MCU 集群，帧短、转发简单。

可选：uint16_t 地址；外层单独 CRC；改名为 cluster envelope。

2. 响应靠什么匹配？

推荐：mesh 层有自己的 seq；mini9P tag 只管文件协议会话。

原因：多跳回程、控制帧和 mini9P payload 都需要统一匹配。

可选：复用 mini9P tag；只在 transport ctx 里维护 pending。

3. 新节点如何收到 ASSIGN？

推荐：REGISTER 上行时中继记录临时反向路径，ASSIGN 用 node_uid 匹配回传。

原因：新节点尚无正式地址，只能靠 UID 和临时路径闭环。

可选：邻居代注册；主机周期广播分配结果。

4. 时钟同步怎么做？

推荐：主机周期发起 TIME_SYNC，用四时间戳估算 offset 和 RTT。

原因：时钟同步是核心目标；成功同步响应可同时作为在线心跳。

可选：先定 10-50 ms 精度；更高精度后续单独设计。

5. 从机转发表放哪里？

推荐：放在从机侧 cluster_link / cluster 模块，由 envelope dispatcher 查询。

原因：转发表属于网络转发，不属于 mini9P server、VFS 或 UART driver。

可选：放入统一 link_service；通过 /sys/routes 只读暴露诊断。

6. cluster_vfs 怎么改？

推荐：从 target/next_hop/client 路由项迁移为 VFS 挂载表。

保留：node_name、m9p_client*、mini9P attach 状态、local_fd -> remote_fid 会话。

移出：next_hop、出端口、全局拓扑、路由可达性，交给 cluster/ 和新链路层。

原因：VFS 只负责用户路径和文件会话；节点事实、调度状态和网络转发应有权威来源。

可选：短期保留兼容字段；未来出现第二种远程文件 backend 后再抽象 backend ops。

7. 第一版控制面做哪些？

推荐：先做 REGISTER、ASSIGN、TIME_SYNC、ROUTE_UPDATE。

原因：TIME_SYNC 作为同步心跳，同时覆盖在线判断、时钟偏移和往返延迟估计。

可选：单独 HEARTBEAT、NODE_INFO、LINK_UP/DOWN、ERROR 延后。
