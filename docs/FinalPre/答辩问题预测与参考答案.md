# 9P-Walkers 答辩问题预测与参考答案

> 本文件整理答辩现场可能被问到的技术问题，覆盖项目整体、协议、主机、从机、扩展能力、实验细节等维度。每题均给出简明参考答案，便于背诵和临场发挥。

---

## 一、项目整体与架构（20 题）

### Q1：请用一句话概括 9P-Walkers 是什么？
**A**：9P-Walkers 是一个面向异构 MCU 集群的实验系统，用 ESP32 作主控、STM32 作节点，通过自定义链路协议和主机间 RPC，把多个 MCU 组织成一个统一的分布式计算与远程诊断平台。

### Q2：项目为什么要叫 9P-Walkers？
**A**：名称中的 9P 指系统采用了类 9P 的精简文件协议 mini9P，用于远端文件和诊断命名空间；Walkers 寓意多个 MCU 节点像步行者一样协同移动/工作。整体强调“用类 9P 协议把 MCU 节点连接起来”。

### Q3：系统的整体架构是怎样的？
**A**：系统有两个网络平面：上层是主机间 IP 平面，ESP32-P4 和 ESP32-S3 通过 mDNS + TCP/9909 + CBOR RPC 互联；下层是 STM32 有线 mesh 平面，通过 UART 1 Mbaud 和自定义 link frame v2 通信。两个平面只在 ESP32 主机汇合，跨主机访问由 owner 主机代理。

### Q4：主机间为什么不直接透传 STM32 link frame？
**A**：因为 STM32 平面使用短地址和本地拓扑，主机间需要更高层的语义（如全局 `mcuN` 命名、拓扑同步、跨主机 read/write）。如果直接透传 link frame，地址和路由都会在主机间冲突。因此由 owner 主机终止 host RPC，再向自己的 STM32 子树发起请求。

### Q5：数据面有哪几种类型？分别用途是什么？
**A**：
- `DATA_MINI9P`：远端文件/诊断，如 `/mcuN/sys/health`。
- `DATA_RPC`：短调用、流式调用、通知、取消。
- `DATA_JOB`：异步计算任务，可查询/取消/重试。
- `DATA_BULK`：大数据传输，当前预留未实现。

### Q6：控制面和数据面如何分离？
**A**：STM32 端使用独立任务和队列：控制面帧进高优先级 `ctrl_tx` 队列，由 mesh_ctrl_task 处理；数据面帧进 `link_tx`/`service_rx` 队列，由 service_task 串行处理。主机端 coordinator_task 只处理控制面和分发数据面，实际 I/O 由 session_manager 异步完成。

### Q7：系统有哪些硬件平台？分别承担什么角色？
**A**：
- ESP32-P4：默认 coordinator、Ethernet 网关、WebShell、Lua 调度、推理。
- ESP32-S3：WiFi STA 主机、host RPC、推理。
- STM32F407：基础 FreeRTOS 节点。
- STM32F429：带 240x320 LCD 的节点，可接收并显示渲染 tile。

### Q8：为什么要用 ESP32 作主控而不是只用 STM32？
**A**：ESP32 有更强的处理能力和网络能力（Ethernet/WiFi），适合运行 TCP/IP、mDNS、HTTP server、Lua 解释器和 LLM 推理运行时；STM32 更适合实时控制、UART mesh 和底层计算。两者分工符合各自优势。

### Q9：项目中的“统一命名空间”是什么？
**A**：`/mcuN/...` 是用户可见的统一路径。例如 `/mcu1/sys/health` 表示第 1 个 MCU 的健康状态文件。cluster_vfs 在主机侧把这个路径映射到对应 STM32 的本地路径 `/sys/health`。

### Q10：项目的使用场景有哪些？
**A**：边缘计算任务分发、远程节点诊断、分布式光线追踪渲染、多主机 MCU 集群管理、以及作为异构嵌入式系统协议栈的研究平台。

### Q11：项目的输入输出是什么？
**A**：输入包括用户命令（WebShell/HTTP）、主机间 RPC 请求、MCU 注册和链路状态上报；输出包括统一文件读取结果、RPC 响应、Job 结果、渲染 tile、节点状态快照等。

### Q12：系统最多支持多少个 STM32 节点？为什么？
**A**：最多 253 个可用短地址。因为 mesh 地址是 8 位，0x00 固定给 host，0xFF 表示未分配/广播，剩余 1~253 可分配给节点。

### Q13：如果节点数量超过 253 怎么办？
**A**：当前版本无法支持。改进方向是把地址扩展为 16 位，或采用分层地址和多域路由。

### Q14：项目中“owner host”是什么意思？
**A**：owner host 是直接通过 UART 管理某个 STM32 子树的 ESP32 主机。每个 STM32 节点只属于一个 owner host，跨主机访问请求由 owner host 代理。

### Q15：系统中“leader”和“coordinator”有什么区别？
**A**：coordinator 是 STM32 平面的控制面权威，负责地址分配和路由计算，运行在 ESP32 上；leader 是主机间平面的协调者，负责全局 `mcuN` 命名和拓扑同步。单主机时两者通常在同一台 ESP32 上。

### Q16：项目的关键约束有哪些？
**A**：
- 热路径不动态分配内存。
- 协议、队列、job 使用固定上限。
- UART RX 每平台只有一个消费者。
- 控制面不能因文件访问、RPC、Job 阻塞。
- 主机间平面只用于可信局域网，无 TLS/鉴权。

### Q17：系统如何保证实时性？
**A**：通过 FreeRTOS 任务优先级、独立队列、增量计算调度、ISR 只搬运字节不处理协议逻辑等方式，保证控制面和高优先级任务不被长计算或文件 I/O 阻塞。

### Q18：项目的代码量大概是多少？
**A**：核心手写代码约 2~3 万行（共享协议 + 主机模块 + 从机模块），加上 CubeMX 生成代码、FreeRTOS、Lua 解释器、llama.cpp 等第三方代码会更大。

### Q19：项目经历过重构吗？
**A**：是的。文档 `docs/refactor_plan.md` 和 `docs/logs/refactor/README.md` 记录了重构过程。旧版 mesh envelope 已删除，当前主线是 link frame v2 + mesh2 + typed pending。

### Q20：项目的最大创新点是什么？
**A**：在资源受限的异构 MCU 上实现了完整的控制面/数据面分离、统一命名空间、多跳 mesh、主机间协作和可扩展计算调度，且大量模块使用固定内存和可测试设计。

## 二、链路层与协议（25 题）

### Q21：link frame v2 的总长度是多少？
**A**：总长度 = 19 字节头部 + payload_len，payload 最大 512 字节，所以单帧最大 531 字节。

### Q22：link frame 头部包含哪些字段？
**A**：magic(2)、version(1)、hdr_len(1)、type(1)、flags(1)、src(1)、dst(1)、ttl(1)、seq(2)、ack(2)、payload_len(2)、hdr_crc(2)、payload_crc(2)。

### Q23：为什么用双层 CRC？
**A**：头 CRC 覆盖头部字段，可以在收齐 19 字节头部后立即判断头部是否损坏；payload CRC 只覆盖 payload，确保数据完整性。双层设计能尽快丢弃坏帧并保留 magic 前缀恢复同步。

### Q24：CRC 用的是什么算法？
**A**：CRC-16/CCITT-FALSE，实现在 `pwos-shared/link/pwos_link_crc.c`。

### Q25：magic 为什么是 "MH"？
**A**：是项目自定义的帧同步前缀，'M' 和 'H' 两个字节用于 parser 在字节流中快速定位候选帧头。

### Q26：流式 parser 的状态机有几个关键状态？
**A**：状态隐含在 `parser->len` 中：len=0 找 magic0；len=1 等 magic1；2~18 收头部；len=19 且 want_len 已计算则收 payload；len=want_len 时最终解码。

### Q27：parser 如何处理噪声？
**A**：未锁定帧头时噪声字节静默丢弃；锁定帧头后不在 payload 中重新同步；头部阶段出错后尽量保留尾部 magic 前缀（1 或 2 字节）以便快速恢复。

### Q28：TTL 字段作用是什么？在哪里递减？
**A**：TTL 用于多跳防环，每转发一次递减 1。在 `pwos-slave/User/mesh2/node_control.c` 的 `forward_view()` 中递减，TTL <= 1 时丢弃。

### Q29：特殊地址 0x00 和 0xFF 分别代表什么？
**A**：0x00 是 host/根节点地址，0xFF 是未分配/广播地址。

### Q30：mesh2 控制面有哪些消息？
**A**：NODE_REGISTER、ADDR_ASSIGN、LEASE_RENEW、LEASE_ACK、LINK_STATE、ROUTE_UPDATE、HOST_ADVERTISE、TIME_SYNC、CTRL_ERROR。

### Q31：NODE_REGISTER 包含哪些字段？
**A**：uid[3]、boot_id、caps、upstream_port，共 24 字节。

### Q32：ADDR_ASSIGN 包含哪些字段？
**A**：uid[3]、boot_id、lease_epoch、lease_ms、addr、flags，共 28 字节。

### Q33：LINK_STATE 的 metric 怎么算？
**A**：当前 metric 固定为 1 表示直连链路。coordinator 收到后生成路由时 metric = link.metric + 1。

### Q34：ROUTE_UPDATE 中的 route_version 有什么用？
**A**：节点端用 route_version 判断路由是否更新，旧 version 忽略，防止乱序或重复的路由更新覆盖新路由。

### Q35：为什么 mesh2 消息用固定长度？
**A**：固定长度消息便于 MCU 解析，无需动态长度计算和内存分配，符合资源受限场景。

### Q36：mini9P 和完整 9P 有什么区别？
**A**：mini9P 是精简版 9P，只支持 attach、walk、open、read、write、stat、clunk，且帧格式和字段做了简化，适合 MCU 和 UART 小帧传输。

### Q37：mini9P 的 qid 是什么？
**A**：qid 是每个文件对象的唯一标识，包含 type、version、object_id，用于区分文件、目录、设备等。

### Q38：主机路径如何映射到节点本地路径？
**A**：`/mcuN/sys/health` 中的 `/mcuN` 部分被 cluster_vfs 去掉，剩下的 `/sys/health` 作为 mini9P 的 walk/open/read 路径发往对应节点。

### Q39：RPC 帧格式包含哪些字段？
**A**：kind、flags、call_id、status、deadline_ms、service、method、payload，头部 16 字节。

### Q40：Job 帧格式包含哪些字段？
**A**：kind、state、kernel、request_id、status、job_id、progress_permille、result_len、payload，头部 20 字节。

### Q41：RPC 流式响应如何保证顺序？
**A**：每个 chunk 的 status 字段承载从 0 开始的序号。session_manager 的 `deliver_data_part` 校验序号连续性，不连续则标记错误。

### Q42：Job 有哪些状态？
**A**：EMPTY、CREATED、QUEUED、ASSIGNED、RUNNING、DONE、FAILED、CANCELLED、LOST。

### Q43：Job 的 LOST 状态什么时候出现？
**A**：节点重启、事务超时或远端任务丢失时，主机把本地 job 标记为 LOST。retry 会创建新 job，不复活旧 boot 上的远端 ID。

### Q44：host RPC 和 MCU RPC 有什么区别？
**A**：MCU RPC 运行在 STM32 平面，封装在 DATA_RPC payload 中；host RPC 运行在 ESP32 主机间 TCP/9909 上，使用 CBOR 编码，不封装进 link frame。

### Q45：host RPC 帧格式有什么特点？
**A**：每帧由 4 字节网络序 CBOR body 长度前缀 + CBOR map 组成，完整 wire frame 上限 1280 字节。

## 三、ESP32 主机控制面（25 题）

### Q46：ESP32-P4 启动流程是什么？
**A**：`app_main()` → 打印芯片信息 → `pwos_coordinator_runtime_start_default()` 初始化 parser/coordinator/session/rpc/job/vfs/UART → 创建 coordinator_task 和 mini9p_probe_task → 启动 LAN、host RPC、HTTP/WebSocket、Lua、inference。

### Q47：coordinator_task 主循环做什么？
**A**：每周期协调控制角色、发送 HELLO、发送 host_advertise、读取 UART 字节并喂给 parser、处理帧、每 5 秒打印一次状态。

### Q48：HELLO 发送周期是多少？
**A**：500ms。

### Q49：host_advertise 发送周期是多少？
**A**：1000ms。

### Q50：状态报告周期是多少？
**A**：5000ms。

### Q51：为什么 follower 要丢弃非 HELLO 的控制帧？
**A**：多主机场景下，只有 leader 才是 STM32 平面的控制面权威。follower 若处理控制帧会导致地址域冲突和脑裂。丢弃非 HELLO 帧保证控制面收敛到单一权威。

### Q52：节点重启后主机如何识别？
**A**：`NODE_REGISTER` 携带 UID 和 boot_id。主机发现 UID 相同但 boot_id 不同，即认为节点重启，会更新 boot_id、lease_epoch，并清理旧 session、fid、pending、job。

### Q53：地址分配算法是什么？
**A**：线性扫描。从 `next_addr` 开始，在 1~253 范围内找第一个未使用的地址，最多尝试 254 次。

### Q54：如果地址池满了会怎样？
**A**：`alloc_addr()` 返回 -1，节点注册失败，不会分配地址。

### Q55：租约默认多长？续期时机？
**A**：默认租约 30 秒。节点在 lease_ms/2 时发起续期，即 15 秒左右。

### Q56：租约续期失败会怎样？
**A**：节点收不到 LEASE_ACK，租约到期后 `node_control` 会回到 OFFLINE 状态，重新选择上游并注册。

### Q57：路由计算是集中式还是分布式？
**A**：集中式。coordinator 根据全局节点表和 LINK_STATE 计算路由，通过 ROUTE_UPDATE 下发给节点。STM32 只维护本地转发表，不运行全局路由算法。

### Q58：正向路由和反向路由分别发给谁？
**A**：假设 local 上报到 peer 的链路。正向路由发给 local，告诉它怎么到 peer；反向路由发给 peer，告诉它怎么到 local。

### Q59：coordinator 如何知道全局拓扑？
**A**：每个 STM32 节点周期上报 `LINK_STATE`，包含本地地址、本地端口、对端地址、对端端口、metric 等信息。coordinator 汇总后知道全网邻接关系。

### Q60：`reconcile_control_role()` 的作用是什么？
**A**：查询 host_rpc 的角色，如果本地从 leader 变成 follower，则清空本地节点表和地址域，避免同一 MCU 同时存在于两个地址域。

### Q61：TX guard 延时是多少？为什么需要？
**A**：2000 微秒。因为 STM32 使用 DMA_NORMAL + ReceiveToIdle，IDLE 回调后需要任务重新 arm DMA，延时避免下一帧落入接收端 rearm 空窗。

### Q62：mini9p_probe_task 做什么？
**A**：周期枚举在线节点，对每个节点读取 `/sys/health`，验证数据面通路并打印健康状态。周期 10 秒，每次只探测一个节点。

### Q63：UART 配置是什么？
**A**：UART1，TX=37，RX=38，波特率 1 Mbaud，8N1，无硬件流控。

### Q64：coordinator 的统计信息包含哪些？
**A**：包含 rx/tx 字节帧数、hello/reg/assign/renew/route 计数、mini9P/RPC/Job 收发数、pending 统计、parse/tx 错误数等。

### Q65：host_coordinator 最大支持多少个节点？
**A**：`PWOS_HOST_COORDINATOR_MAX_NODES` 定义为 32。

### Q66：节点表项包含哪些字段？
**A**：valid、addr、upstream_port、uid[3]、boot_id、lease_epoch、lease_ms。

### Q67：如何处理重复注册？
**A**：UID 和 boot_id 都相同则认为是重复注册，只更新 upstream_port，不重新分配地址。

### Q68：地址复用如何防止？
**A**：每个地址分配前调用 `addr_in_use()` 检查；如果已有节点使用则继续扫描。

### Q69：coordinator 如何选择自己的 UID？
**A**：P4 coordinator 的 UID 是固定的魔术值 + 随机数，如 `0x5034574F`（"P4WO"）、随机、`0x434F4F52`（"COOR"）。

### Q70：多主机下 follower 的 STM32 子树怎么办？
**A**：follower 继续管理自己的 STM32 子树（owner），但地址和 `mcuN` 命名由 leader 统一分配。跨主机访问请求由 owner host 代理。

## 四、主机数据面与会话层（25 题）

### Q71：session_manager 的作用是什么？
**A**：并发管理 mini9P、RPC、Job 的请求/响应匹配，维护 session 和 typed pending 表。

### Q72：sessions 和 pending 数组大小分别是多少？
**A**：sessions 最大 32 个（按节点），pending 最大 8 个（全局）。

### Q73：typed pending 的匹配键是什么？
**A**：`(data_type, src_addr, wire_tag)`。

### Q74：wire_tag 如何生成？
**A**：`alloc_wire_tag_locked()` 递增 `next_wire_tag`，跳过 0，并检查是否与当前 active pending 冲突。

### Q75：为什么 pending 表只有 8 个槽位？
**A**：固定大小避免动态内存，适合资源受限场景。实际并发度受应用层限制，8 个槽位通常足够。

### Q76：如果 pending 满了会怎样？
**A**：返回 `PWOS_SESSION_ERR_QUEUE_FULL`，请求失败。

### Q77：`acquire_client` 和 `release_client` 为什么要成对调用？
**A**：它们之间获取了对应 session 的 client mutex，成对调用保证锁释放，避免死锁。

### Q78：不同节点之间的请求可以并发吗？
**A**：可以。每个节点有独立的 client mutex，不同节点的请求互不影响。

### Q79：节点重启时未完成的 pending 会怎样？
**A**：`reset_node()` 会把该节点所有未完成 pending 标记为 STALE_BOOT 并 signal 唤醒，等待方会收到错误。

### Q80：session_manager 的同步回调由谁提供？
**A**：由具体平台提供。ESP32 端使用 FreeRTOS mutex 和 binary semaphore；PC 测试可以留空并在 send 中同步投递响应。

### Q81：cluster_vfs 的作用是什么？
**A**：把用户路径 `/mcuN/...` 映射到具体 STM32 节点的 mesh 地址和本地路径，管理 fd 到 remote_fid 的映射。

### Q82：cluster_vfs 的 route 包含哪些字段？
**A**：state、target（如 mcu1）、addr、uid[3]、boot_id、generation。

### Q83：generation 字段的作用是什么？
**A**：检测节点地址或 boot_id 变化。如果 route 的 generation 与 file 记录不一致，说明节点身份已变，fd 失效。

### Q84：`resolve_path()` 如何处理离线节点？
**A**：如果匹配到前缀但节点离线，设置 `matched_offline` 标志，继续查找；最终返回 `PWOS_SESSION_ERR_NO_ROUTE`。

### Q85：cluster_vfs 如何防止 fd 指向错误节点？
**A**：open/read/write/close 前后都会检查 file 的 route_generation 是否与当前 route 的 generation 一致，不一致则返回 STALE_BOOT。

### Q86：mini9P client 的 `session_transport` 做了什么？
**A**：为 mini9P client 提供底层传输：分配 pending、替换 wire_tag、发送请求、等待响应、恢复原始 tag。

### Q87：RPC client 如何发送请求？
**A**：`pwos_rpc_client_call()` 构造 RPC 帧，通过 `session_manager_request_data()` 发送，等待响应后解码。

### Q88：RPC 支持哪些 kind？
**A**：REQUEST、RESPONSE、CANCEL、STREAM_CHUNK、STREAM_END。

### Q89：RPC 支持哪些 flags？
**A**：ONEWAY、STREAM。

### Q90：Job manager 支持哪些操作？
**A**：caps、submit、status、result、cancel、retry。

### Q91：Job manager 中 host_job_id 和 remote_job_id 有什么区别？
**A**：host_job_id 是主机本地 job 槽位标识；remote_job_id 是 STM32 远端计算任务的标识。retry 会创建新的 host_job_id 和 remote_job_id。

### Q92：Job 超时如何处理？
**A**：请求有 deadline，超时后 session_manager 释放 pending 并返回 DEADLINE 错误。主机也可以显式发送 CANCEL。

### Q93：集群 VFS 的 `/` 目录 list 会返回什么？
**A**：返回所有在线节点的 `mcuN` 目录项。

### Q94：mini9P server 运行在哪里？
**A**：运行在 STM32 的 service_task 中，处理主机发来的 mini9P 请求，调用 local_vfs 读写文件。

### Q95：session_manager 的默认 deadline 是多少？
**A**：默认 1000ms，可通过配置修改。

## 五、STM32 从机侧（30 题）

### Q96：STM32F407 启动后创建几个任务？分别是什么？
**A**：8 个任务：uart_rx、link_rx、link_tx、port_mgr、mesh_ctrl、service、compute、app、diag。

### Q97：任务优先级如何排列？
**A**：uart_rx 最高（idle+5），link_rx/link_tx/port_mgr/mesh_ctrl 次之（idle+4），service（idle+3），app（idle+2），compute 和 diag 最低（idle+1）。

### Q98：为什么 uart_rx_task 优先级最高？
**A**：它负责尽快重新 arm UART ReceiveToIdle DMA，防止接收中断丢失后 RX 停住。

### Q99：link_rx_task 的职责是什么？
**A**：从 link_rx 队列接收 ISR 投递的帧块，先交给 port_manager 处理链路维护帧；若不是则投递到 mesh_rx 队列。

### Q100：link_tx_task 为什么优先发送 ctrl_tx？
**A**：控制面帧优先级更高，保证注册、租约、路由等控制消息优先传输。

### Q101：发送完成后为什么要 delay 2ms？
**A**：给对端 DMA rearm 留出空窗，避免下一帧撞上 IDLE 回调后的重新 arm 窗口。

### Q102：frame_pool 是什么？
**A**：固定大小的帧块池，每个块包含完整 link frame 缓冲区。所有数据面/控制面收发都从这里分配和归还，避免动态内存。

### Q103：port_manager 的端口状态有哪些？
**A**：DISABLED、PROBING、LINK_UP、HOST_CANDIDATE、UPSTREAM、PEER、SUSPECT、QUARANTINED。

### Q104：选择上游端口的优先级是什么？
**A**：已标记 UPSTREAM > HOST_CANDIDATE > 声明 UPSTREAM_REACHABLE 的 PEER。

### Q105：HELLO 发送周期是多少？
**A**：500ms。

### Q106：端口多久没收到帧会进入 SUSPECT？
**A**：2000ms。

### Q107：node_control 的节点状态有哪些？
**A**：OFFLINE、REGISTERING、ASSIGNED、LIMITED。

### Q108：节点如何发送注册请求？
**A**：`node_control_tick()` 在 OFFLINE 状态时选择上游端口，调用 `send_register()` 发送 CTRL_NODE_REGISTER。

### Q109：收到 ADDR_ASSIGN 后节点做什么？
**A**：如果 UID/boot_id 匹配自己，保存 addr、upstream_port、lease_epoch、lease_ms，状态变为 ASSIGNED，清空路由表，通知 port_manager。

### Q110：节点如何 relay 下游节点的注册？
**A**：收到 NODE_REGISTER 且本机已 ASSIGNED 时，记录 pending_register，并向上游端口转发该注册帧。

### Q111：收到给下游节点的 ADDR_ASSIGN 时怎么办？
**A**：从 pending_register 中找到对应条目，设置到该下游地址的直连路由，并转发 ADDR_ASSIGN。

### Q112：路由表最多多少条？
**A**：`PWOS_NODE_CONTROL_MAX_ROUTES` 定义为 16。

### Q113：TTL 在哪里递减？
**A**：在 `forward_view()` 中，`view->ttl - 1`。TTL <= 1 时不转发。

### Q114：HOST_ADVERTISE 如何选择 authority？
**A**：按 `(epoch, priority, host_uid)` 比较。只有新 leader 更好，或同一 leader 但 TTL 更大（路径更短）时才切换，避免三角环抖动。

### Q115：service_runtime 接收哪些数据面帧？
**A**：DATA_MINI9P、DATA_RPC、DATA_JOB。只处理目的地址是本机的帧。

### Q116：local_vfs 暴露了哪些路径？
**A**：
- `/sys/{health,tasks,ports,links,neighbors,routes,sessions,queues,log,build,fault}`
- `/compute/{caps,load,jobs}`
- F429 还有 `/display/{status,tile}`

### Q117：service_runtime 为什么是串行处理？
**A**：避免多个请求同时修改 server 状态和 local_vfs，简化并发控制。

### Q118：rpc_service 内置了哪些服务？
**A**：`system.ping`、`system.stream`、`system.info`、`system.notify`、`system.delay`、`system.fail`。

### Q119：job_service 如何与 compute_worker 交互？
**A**：job_service 解析 Job 请求，调用 compute_worker 的 submit/cancel/get_result 等接口。

### Q120：compute_worker 支持哪些 kernel？
**A**：hash、vector_add、matmul、mandelbrot、raytrace_tile。

### Q121：compute_worker 有多少个 job 槽位？
**A**：`PWOS_COMPUTE_MAX_JOBS` 在头文件中定义。

### Q122：增量计算如何调度？
**A**：compute_task 每周期调用 `pwos_compute_worker_step()`，执行一个 job 的一个 work unit，然后 `vTaskDelay(1ms)` 让出 CPU。

### Q123：hash kernel 用的是什么算法？
**A**：FNV-1a 32 位哈希。

### Q124：matmul kernel 的矩阵尺寸限制是多少？
**A**：每个维度最多 8。

### Q125：Mandelbrot kernel 使用什么数据类型？
**A**：使用 16.16 定点数（Q16）进行迭代计算。

## 六、主机间网络与扩展能力（25 题）

### Q126：host RPC 用什么端口？
**A**：TCP/9909。

### Q127：host RPC 帧格式有什么特点？
**A**：4 字节网络序 CBOR body 长度前缀 + CBOR map，完整帧上限 1280 字节。

### Q128：为什么用 CBOR？
**A**：CBOR 是二进制 JSON，体积小、解析简单、适合嵌入式和网络传输。

### Q129：mDNS 服务类型是什么？
**A**：`_pwos._tcp`，发布 TCP/9909 服务。

### Q130：leader election 的比较键是什么？
**A**：按 `(epoch, priority, host_uid)` 字典序比较。epoch 优先级最高，其次是 priority，最后是 host_uid。

### Q131：epoch 是什么？
**A**：主机代次，持久化在 NVS 中，启动时递增。epoch 越大表示主机实例越新。

### Q132：follower 如何知道 leader？
**A**：通过 host_rpc 的对端发现机制和 election 状态查询。leader 也会通过 HOST_ADVERTISE 向 STM32 平面通告。

### Q133：如果 leader 宕机会怎样？
**A**： follower 在超时后重新选举。新的 leader 会重新分配全局 `mcuN` 名称。

### Q134：跨主机 read/write 的流程是什么？
**A**：请求发到 owner host 的 host RPC，owner host 在自己的 STM32 子树中发起 mini9P 请求，结果通过 host RPC 返回。

### Q135：WebShell 提供哪些命令？
**A**：`ls`、`cat`、`rpc`、`job`、`hosts`、`net status`、`fault` 等。详见 `pwos-master-esp32p4/README.md`。

### Q136：HTTP server 和 WebSocket console 的作用？
**A**：HTTP server 提供 Web 访问入口；WebSocket console 提供类似串口的交互式命令行。

### Q137：Lua 渲染调度器做什么？
**A**：枚举在线 MCU，把 240x320 的 smallpt 场景拆成 16x7 的 tile，每个 tile 作为一个 raytrace_tile Job 提交给节点，完成后把 RGB565 tile 写入 F429 的 `/display/tile`。

### Q138：渲染参数是什么？
**A**：image 240x320，tile 16x7，samples 4，max depth 4。

### Q139：本地 LLM 推理运行时当前能力如何？
**A**：当前是实验性骨架，P4/S3 上有 `inference_runtime` 和 `dist_inference_service`，但完整功能仍在完善。

### Q140：Lab4 用了什么模型？
**A**：Qwen2.5-7B-Instruct，量化格式 Q4_K_M。

### Q141：Lab4 测量的指标有哪些？
**A**：加载时间、首 Token 延迟（TTFT）、prefill 吞吐、decode 吞吐、峰值内存、输出质量。

### Q142：Lab4 最优配置是什么？
**A**：`--threads 24~32` + `--no-mmap`（内存允许时），prefill 约 87 t/s，decode 约 14 t/s。

### Q143：Lab4 RPC 分布式推理的拓扑？
**A**：一台主机加载模型并作为 head，另一台机器运行 rpc-server 作为 worker，通过局域网 TCP 连接。

### Q144：Lab4 RPC 推理性能如何？
**A**：轻量 bench 下 prefill 略低于单机（网络传输开销），decode 受网络噪声影响大。RPC 更适合内存扩展而非普通局域网加速。

### Q145：Lab4 Ray 做了什么？
**A**：用 Ray task 把多个 prompt 分发到多个 `llama-server` HTTP 服务，实现批量推理、并发压力测试和失败重试。

### Q146：Lab4 并发压力测试结果？
**A**：并发 1/2/4 时总耗时从 167s 降到 71s，吞吐提升，但平均延迟和 P95 延迟也上升，因为 CPU 资源竞争。

### Q147：Lab4 失败重试如何实现？
**A**：请求首次失败（如端口未启动）后，立即尝试下一个 server，直到成功或全部失败。实验中 4 个请求最终全部成功。

### Q148：Lab4 和主项目是什么关系？
**A**：Lab4 是独立的 Linux 分布式推理实验，验证 llama.cpp 和 Ray；主项目的推理服务骨架可借鉴 Lab4 经验进一步完善。

### Q149：主机间通信有安全机制吗？
**A**：当前没有 TLS 或鉴权，只部署在可信隔离 LAN 中。改进方向是添加预共享密钥或证书。

### Q150：为什么主机间平面要独立于 STM32 平面？
**A**：两个平面的 scale、介质、协议都不同。主机间需要高带宽 IP 和复杂语义；STM32 平面需要低延迟 UART 和简单协议。分离后各自优化，也便于多主机扩展。

## 七、设计思想与工程实践（20 题）

### Q151：为什么选择 FreeRTOS 而不是裸机？
**A**：FreeRTOS 提供任务调度、队列、信号量，便于实现控制面/数据面分离、增量计算、并发等待等机制。

### Q152：为什么选择静态任务和静态内存？
**A**：避免动态内存分配带来的碎片、失败和不可预测延迟，适合资源受限和对实时性有要求的嵌入式场景。

### Q153：控制面不能阻塞的原则如何体现？
**A**：控制面任务只做短平快的状态处理；文件 I/O、RPC、Job 都走 pending/队列；长计算在低优先级任务中增量执行。

### Q154：项目中使用了哪些设计模式？
**A**：状态机（parser、port_manager）、观察者/回调（session_manager 同步回调）、代理模式（cluster_vfs 映射）、生产者-消费者（队列）。

### Q155：如何保证模块边界清晰？
**A**：每个模块只拥有一种状态：link 层只负责帧格式，mesh2 只负责控制 payload，mini9P 只负责文件语义，port_manager 只负责端口发现，node_control 只负责地址/路由。

### Q156：异常处理有哪些原则？
**A**：CRC 错丢弃单帧；无路由返回 E_NO_ROUTE；TTL 耗尽丢弃；boot_id 变化清理旧状态；pending 超时释放；不 catch 空异常。

### Q157：项目中如何做测试？
**A**：共享协议和各主机/从机模块都有独立的 CMake 测试工程，构建到 `/tmp`，用 ctest 运行。mini9P client 还有轻量 host test。

### Q158：如何保证不同平台代码复用？
**A**：把与 HAL/IDF 无关的协议和算法放在 `pwos-shared/` 中；平台相关部分通过回调、配置结构体注入同步和 I/O 机制。

### Q159：项目中如何避免类型不安全？
**A**：使用 C11 标准类型（stdint.h）、枚举、结构体，避免 `void*` 滥用和魔术数字；没有使用 `as any` 或 `@ts-ignore` 这类类型抑制。

### Q160：代码中有哪些性能优化？
**A**：帧解析 O(n) 线性扫描、固定内存池、DMA 接收、控制面高优先级队列、增量计算、头部 CRC 早失败。

### Q161：项目中如何保证可扩展性？
**A**：数据面类型可扩展（0x80~0x8F）、kernel 可扩展、host RPC 方法可扩展、新的诊断文件只需在 local_vfs 中添加路径处理。

### Q162：如果要在系统中加入新的数据类型，需要改哪些地方？
**A**：在 `pwos_link_frame.h` 定义新 type；session_manager 中添加对应 deliver 路径；STM32 service_runtime 中处理新 type；可能需要在共享目录新增协议模块。

### Q163：如果要在 compute_worker 加入新 kernel，需要改哪些地方？
**A**：在 `pwos_job_protocol.h` 添加 kernel 枚举；在 `compute_worker.c` 添加 `validate_input` 分支、`prepare_result` 分支、`step_xxx` 函数；在 `pwos_job_kernel_name()` 中添加名称。

### Q164：项目中使用了哪些第三方库？
**A**：ESP-IDF、FreeRTOS、STM32 HAL、Lua 5.4、llama.cpp、CBOR（可能 TinyCBOR 或自实现）、mDNS（ESP-IDF 组件）。

### Q165：为什么不直接用 MQTT/CoAP 而自定义协议？
**A**：项目目标是研究和实现一个面向 MCU 集群的统一协议栈，自定义协议可以精确控制帧大小、状态机、路由和命名空间，便于实验和创新。

### Q166：项目中最复杂的模块是哪个？
**A**：session_manager 和 cluster_vfs 较复杂，因为它们要处理并发、pending 匹配、路径映射、generation 一致性等问题。

### Q167：项目中最容易出现 bug 的地方是什么？
**A**：parser 重同步、pending 超时释放、多主机角色切换时的状态清理、compute_worker 的并发访问。

### Q168：你们是如何调试这个系统的？
**A**：串口日志、GDB 观察点、`/sys/*` 诊断文件、WebShell 命令、PC 单元测试、上板验收脚本。

### Q169：项目的代码组织有什么特点？
**A**：按角色分层：共享协议、主机模块、从机模块、实验目录，每层内部再按功能分子目录。

### Q170：如果时间充裕，最想改进哪个部分？
**A**：可回答：地址空间扩展、主机间安全、BULK 分片、Job 优先级调度、分布式推理服务完善。

## 八、临场应变与主观问题（10 题）

### Q171：你们的贡献是什么？
**A**：按个人实际负责模块回答。例如：我负责 xxx 模块的设计/实现/调试，解决了 xxx 问题，编写了 xxx 测试，验证了 xxx 功能。

### Q172：项目难点在哪里？
**A**：难点包括：
- 在资源受限设备上实现可靠的多跳通信；
- 控制面与数据面的隔离与并发调度；
- 多主机场景下的一致性和脑裂防护；
- 长计算不阻塞实时控制面。

### Q173：与同类项目相比有什么优势？
**A**：
- 完整的异构 MCU 集群协议栈，而非单一通信库；
- 统一的 `/mcuN` 命名空间；
- 控制面/数据面严格分离；
- 多主机协作能力；
- 零动态内存设计；
- 充分的 PC 单元测试。

### Q174：项目有什么不足？
**A**：
- 地址空间只有 8 位；
- 路由 metric 仅跳数；
- 无 BULK 分片；
- 主机间无安全；
- 推理服务较实验性。

### Q175：如果评委质疑“这个项目有什么用”，怎么回答？
**A**：强调这是面向边缘计算和分布式嵌入式系统的研究平台，可用于工业传感器集群、分布式诊断、边缘 AI 推理、教学实验等。重点是验证协议和架构，而非直接商用。

### Q176：如果评委说“LLM 在 ESP32 上跑不动”，怎么回答？
**A**：当前 LLM 推理是实验性骨架，Lab4 在 Linux 上验证了 llama.cpp 分布式推理；ESP32 端更多是接口和调度，实际推理可 offload 到更强设备或通过 RPC 扩展。

### Q177：如果评委问“为什么不用现成的 ROS/ Zephyr/ RT-Thread”，怎么回答？
**A**：项目目标是研究和实现自定义的轻量级协议栈，聚焦异构 MCU 的统一命名、多跳 mesh 和主机间协作。使用现有 OS 会隐藏这些核心问题，不利于协议层面的创新和教学。

### Q178：如果评委问“数据面性能怎么样”，怎么回答？
**A**：UART 1 Mbaud 下，单帧最大 531 字节，理论最大吞吐约 100KB/s 左右；实际受协议开销、DMA 重 arm、任务调度影响。当前重点是功能正确性，性能优化是后续方向。

### Q179：如果评委问“如何保证节点间的同步”，怎么回答？
**A**：节点间不追求全局时钟同步，而是通过 lease 周期续期、route_version、boot_id 等机制保证状态一致性。时间同步目前是预留字段。

### Q180：如果评委问“你们五个人是怎么分工的”，怎么回答？
**A**：按成员 A~E 的分工简要说明：A 负责底层协议，B 负责主机控制面，C 负责数据面/会话层，D 负责 STM32 从机，E 负责主机间网络和扩展能力。强调交叉讨论和模拟 Q&A。

## 十、答辩现场建议

1. **遇到不会的问题**：诚实回答“这部分我目前没有深入研究”，但可以补充相关模块负责人的名字或说明思路，不要编造。
2. **被追问代码细节**：直接引用文件路径和函数名，显得熟悉代码。
3. **时间控制**：40 分钟 PPT 平均每页 1~1.5 分钟，重点页可讲到 3 分钟。
4. **互相补位**：如果某人被问到不熟悉的领域，其他成员可以礼貌补充。
5. **保持冷静**：评委追问细节是正常环节，不是刁难，展现思考过程比完美答案更重要。
