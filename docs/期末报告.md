## 1. 项目概述

9P-Walkers 是一个面向异构 MCU 集群的轻量级分布式实时操作系统原型。项目以 ESP32-P4
和 ESP32-S3 作为主控节点（Host），以 STM32F407 和 STM32F429 作为从机节点（Slave），
通过 UART 物理链路和自定义轻量协议栈，将多个 MCU 组织为一个统一命名空间下的分布式系统。

**核心设计理念**源自 Plan 9 操作系统的"一切皆文件"思想：将外设状态、诊断信息、计算
能力统一抽象为文件，通过 mini9P 远程文件协议实现跨节点透明访问。用户最终面对的是
`/mcuN/...` 形式的统一路径，而非分散的设备寄存器或专用控制接口。

**主要功能特性**：

- **统一命名空间**：通过 `cluster_vfs` 将多个从机的文件树挂载到 `/mcuN/...` 路径下
- **mini9P 远程文件协议**：支持 attach、walk、open、read、write、stat、clunk 七种基本操作
- **MCU RPC**：支持 unary 调用、streaming、oneway 通知、cancel 和 deadline 超时
- **异步 Job System**：支持 hash、vector add、matmul、Mandelbrot、smallpt 光线追踪
- **分布式光线追踪渲染**：P4 Lua 调度器将 240×320 场景拆分为 tile 分发给多个 STM32 并行计算
- **本地 LLM 推理**：基于 llama2.c 移植的 Transformer 推理引擎，运行在 ESP32-S3/ESP32-P4 上
- **多主机协同**：ESP32-P4 (Ethernet) 与 ESP32-S3 (WiFi) 通过 host RPC 实现跨主机节点访问
- **WebShell**：基于 HTTP/WebSocket 的远程命令行交互界面
- **故障注入与可观测性**：完整的 `/sys` 诊断命名空间、运行时统计和 Debug 故障注入

## 2. 系统最终设计

### 2.1 整体架构

系统包含两个相互独立、只在 ESP32 主机汇合的网络平面：

```text
┌────────────────── 主机间 IP 平面 ──────────────────┐
│  ESP32-P4 (Ethernet)  <-->  router  <-->  ESP32-S3 (WiFi)  │
│  mDNS discovery + TCP/9909 + bounded CBOR host RPC          │
└─────────────────┬─────────────────────┬─────────────────────┘
                  │ UART 1 Mbaud        │ UART 1 Mbaud
┌─────────────────▼─────────────────────▼─────────────────────┐
│                 STM32 有线 mesh 平面                         │
│    link frame v2 + mesh2 control + typed data plane         │
│    节点可作为 relay，支持多 UART、多跳转发                    │
└─────────────────────────────────────────────────────────────┘
```

**关键设计决策**：

1. **逻辑中心式、物理链路可演进**：主机维护全局命名空间和拓扑，STM32 仅保存本地转发表，
   不运行全局路由算法。物理上支持链式多跳中继，但命名空间主导权始终在主机侧。

2. **两平面隔离**：主机间不透传 STM32 link frame。跨主机访问由 owner 主机终止 host RPC，
   再在自己的 STM32 子树中发起 mini9P/RPC/Job 请求。

3. **热路径零动态分配**：所有协议帧、队列、Job 槽位均使用固定上限的静态内存，避免
   MCU 环境下的内存碎片化和分配延迟。

![双平面Mesh架构](FinalPre/dual_plane_mesh.svg)

**系统启动流程**：ESP32 主机与 STM32 从机并行启动，主机依次完成芯片初始化、Coordinator Runtime、网络服务和用户服务；从机依次完成 HAL 外设初始化、FreeRTOS 对象创建和调度器启动。

![系统启动序列](FinalPre/boot_sequence.svg)

### 2.2 STM32 从机平面

![STM32节点架构](FinalPre/slave_arch.svg)

#### 2.2.1 任务架构

STM32 节点（F407/F429）运行 FreeRTOS，采用严格分层的任务流水线：

```text
UART DMA callback (ISR)
  --> link_rx queue
  --> link_rx_task         (优先级: 高)
       --> port_manager    (HELLO FSM、peer 身份管理)
       --> mesh_rx queue
  --> mesh_ctrl_task       (优先级: 高)
       --> node_control    (地址、lease、路由、中继转发)
       --> service_rx queue
  --> service_task         (优先级: 中)
       --> mini9P server   --> local_vfs
       --> rpc_service
       --> job_service     --> compute_worker
  --> compute_task         (优先级: 低)
       --> compute_worker  (增量 kernel 执行)

ctrl_tx / link_tx queue --> link_tx_task --> UART DMA TX
```

![增量调度时间线](FinalPre/incremental_compute.svg)

**关键不变量**：

- ISR 只搬运字节、更新 DMA 状态并通知任务，不执行任何协议逻辑
- UART RX 只有 `link_rx_task` 一个消费者
- 控制队列（ctrl_tx）优先于数据队列（link_tx）
- `service_task` 只做短时协议解析，不等待 compute 完成，因此 HELLO、lease、route 等
  控制面在计算期间不受影响
- 任一端口错误不能停止其他端口
- 帧块入队成功后所有权转移，所有失败路径必须归还块

#### 2.2.2 控制面 Bootstrap

节点启动后的自组网流程：

![节点加入状态机](FinalPre/node_join.svg)

1. 各端口周期发送 `LINK_HELLO`，携带 UID（3 字节硬件标识）、boot_id、port_id 和 capability
2. 收到 `LINK_HELLO_ACK` 后，端口进入 PEER 或 HOST_CANDIDATE 状态
3. 未分配节点选择 coordinator（直连主机）或可到达 upstream 的 relay peer
4. 通过 `CTRL_NODE_REGISTER` 注册（可由 relay 向上游转发）
5. Coordinator 返回 `CTRL_ADDR_ASSIGN`，分配短地址和 lease 有效期
6. 节点周期上报 `CTRL_LINK_STATE`（邻接关系和链路质量）
7. Coordinator 计算全局拓扑，下发带 version 的 `CTRL_ROUTE_UPDATE`
8. 节点进入 lease 续期循环

**安全约束**：节点只接受当前 authority/upstream 的控制指令；旧 route version 不得覆盖
新版本；boot ID 改变后所有旧状态（session、fid、RPC pending、Job）全部失效。

#### 2.2.3 本地命名空间

```
/sys/
  health      只读  节点综合健康状态
  tasks       只读  各 FreeRTOS 任务的 heartbeat 和 stack watermark
  ports       只读  UART DMA 统计（rx/tx bytes, frames, errors）
  links       只读  直连 peer 列表
  neighbors   只读  邻居节点信息
  routes      只读  当前本地转发表
  sessions    只读  活跃 session 统计
  queues      只读  frame pool 使用率和高水位
  log         只读  最近日志
  build       只读  固件版本和构建时间
  fault       读写  Debug 故障注入控制（仅 Debug 构建）
  info        只读  节点 UID、boot_id、地址
  uart        只读  各 UART 端口详细统计

/compute/
  caps        只读  支持的 kernel 列表
  load        只读  当前计算负载
  jobs        只读  活跃 job 列表

/display/     仅 F429
  status      只读  显示屏状态
  tile        只写  写入 tile 像素数据（RGB565）
```

#### 2.2.4 模块所有权

| 模块 | 职责 | 所属状态 |
|------|------|----------|
| `uart_dma_port` | HAL UART/DMA、parser、rearm、TX complete、硬件统计 | 各物理端口 |
| `frame_pool` | 固定大小帧块的生命周期管理 | 全局 |
| `port_manager` | HELLO FSM、peer UID/boot/capability、端口健康 | 各物理端口 |
| `node_control` | 本机地址、lease、authority、route、relay 和数据转发 | 全局 |
| `service_runtime` | typed data 分发、mini9P server | 全局 |
| `rpc_service` | MCU RPC 方法和延期/流式调用状态 | 全局 |
| `job_service` | 远端 job 协议、静态 job 槽管理 | 全局 |
| `compute_worker` | 低优先级增量 kernel 执行 | 全局 |
| `local_vfs` | 路径解析、权限检查、qid 分配、运行时文本回调 | 全局 |
| `fault_control` | Debug 故障注入（丢包/损坏/延迟/端口down/recover） | 全局 |

### 2.3 ESP32 主机内部设计

![ESP32主机架构](FinalPre/master_arch.svg)

#### 2.3.1 主机分层架构

```text
UART RX task (唯一消费者)
  --> coordinator_runtime
       --> host_coordinator      地址分配、lease 管理、拓扑维护、路由计算
       --> session_manager       typed pending、全局 tag 分配、deadline 管理
       --> cluster_vfs           /mcuN 路径映射、fid 管理、generation 版本
       --> slave_rpc             DATA_RPC 客户端
       --> job_manager           DATA_JOB 生命周期管理

WebShell / Lua / inference
  --> host_shell / host_api / host_jobs
  --> 上述主机服务（通过回调接口）
```

**ESP32-P4 特有功能**：Ethernet (IP101 PHY)、HTTP Server、WebSocket、Lua 渲染调度器、
LLM 推理引擎、host RPC server/client。

**ESP32-S3 特有功能**：WiFi STA、LLM 推理引擎（主要运行平台）、host RPC client。

**共享模块**：coordinator_runtime、host_coordinator、session_manager、slave_rpc、
job_manager、cluster_vfs、host_shell 等通过编译宏在 P4/S3 间复用。

#### 2.3.2 Coordinator Runtime

Coordinator 是主机侧最核心的模块，负责：

- **UART 驱动**：初始化 ESP-IDF UART driver，单一 RX task 消费字节流
- **Link Parser**：驱动 `pwos_link_parser` 增量解析帧
- **控制帧处理**：在同一任务中处理 HELLO、REGISTER、LINK_STATE 等控制帧
- **Typed Data 分发**：按 `(data_type, src_addr, tag)` 匹配 pending，唤醒等待者
- **TX 互斥**：使用 mutex 和帧间 guard 防止多上层并发写交错

**关键设计约束**：

- UART RX 只有一个消费者，请求发起方通过 session manager 的 semaphore 等待结果
- 控制队列优先于数据队列
- 不直接在 UART RX task 中执行用户回调

![关键分发模式](FinalPre/code_snippet.svg)

#### 2.3.3 Session Manager

Session Manager 实现 typed pending 机制：

- Pending 键为 `(data_type, src_addr, wire_tag)`，确保 mini9P、RPC、Job 三类响应不会串线
- 全局 wire tag 单调递增，支持多路并发
- 节点 boot ID 改变时，清理该节点所有旧 session、fid 和 pending
- 本地超时后自动发送 CANCEL 并释放 pending

![Typed Pending 匹配流程](FinalPre/pending_match.svg)

![Typed Pending 表结构](FinalPre/pending.svg)

#### 2.3.4 Cluster VFS

`cluster_vfs` 实现统一命名空间的核心逻辑：

![统一命名空间路径映射](FinalPre/path_mapping.svg)

- 将 `/mcuN/sys/health` 映射为 `(target_node, /sys/health)` 的 mini9P 请求
- 管理 fid 到节点的映射和 generation 版本控制
- 支持 `ls /` 的根目录聚合（合并所有在线节点的顶层目录）
- 提供 `walk_path`、`open_path` 等便捷接口

#### 2.3.5 Host Shell / WebShell

命令系统共支持 15 条命令，通过 `command_service` 统一分发：

- **文件操作**：`cat`、`ls`、`stat`、`echo`
- **RPC 命令**：`rpc`、`stream`、`notify`（支持 deadline 参数）
- **Job 命令**：`job submit/status/result/cancel/list/caps/retry`
- **LLM 命令**：`llm`（本地/远程推理，提交→轮询→读取）
- **故障注入**：`fault status/drop/corrupt/delay/down/recover/clear/reboot-self`
- **系统命令**：`hosts`、`net status`

WebShell 架构：

- HTTP `/` 返回嵌入式前端页面（不依赖 CDN）
- WebSocket `/ws` 提供实时双向命令通道
- 每条命令记录 `(socket_fd, generation)`，断线或 fd 复用后旧结果直接丢弃
- 命令执行在独立 worker 中进行，不阻塞 WebSocket 事件循环

### 2.4 主机间 IP 平面

![多主机协同](FinalPre/host_rpc.svg)

#### 2.4.1 物理网络

- ESP32-P4：IP101 Ethernet PHY，DHCP 获取 IP，mDNS 发布 `pwos-xxxx.local`
- ESP32-S3：WiFi STA 模式，默认 SSID/密码为 `pwos-network/pwos-network`
- 两台主机通过 UART 分别管理各自的 STM32 子树
- STM32 link frame 不在 IP 网络中透传

#### 2.4.2 发现与选主

- 每台主机从 MAC 地址派生稳定 `host_uid`（6 字节）和 hostname
- mDNS 发布 `_pwos._tcp` 服务，端口 9909
- NVS 持久化 host epoch（每次启动递增）
- 选主算法：按 `(epoch, priority, host_uid)` 字典序排序
  - P4 默认 priority = 300，S3 默认 priority = 200
  - epoch 越大越优先（重启次数多的主机更稳定）
- Peer 15 秒未刷新后过期，从成员表移除并重新收敛
- Leader 汇总各 owner 的节点快照，分配全局 `mcuN` 名称

#### 2.4.3 Host RPC 协议

基于 TCP/9909，使用 4 字节大端长度前缀 + bounded CBOR map：

- **帧格式**：长度前缀（4B 网络序）+ 规范 CBOR map（≤1024B payload），总上限 1280B
- **方法**：`host.advertise`、`topology.whoowns`、`topology.sync`、`cluster.read_node`、
  `cluster.write_node`、`inference.*`（分布式推理预留）
- **约束**：拒绝重复字段、非规范整数和不定长 CBOR；无 TLS/鉴权（仅可信隔离 LAN）

#### 2.4.4 跨主机访问流程

1. 用户在 P4 WebShell 执行 `cat /mcu2/sys/health`
2. P4 的 `cluster_vfs` 发现 mcu2 属于 S3（通过全局拓扑表）
3. P4 通过 host RPC 向 S3 发送 `cluster.read_node` 请求
4. S3 的 coordinator 向自己的 STM32 子树发起真实 mini9P 请求
5. S3 将 mini9P 响应通过 host RPC 返回给 P4
6. P4 将结果返回给用户

**Alpha 限制**：跨主机主要支持 read/write；远端目录 list/stat 尚未完整代理；
固定容量适合控制和诊断流量，不适合大文件或视频流。

### 2.5 协议栈设计

![端到端数据流](FinalPre/dataflow.svg)

#### 2.5.1 Link Frame v2（数据链路层）

定义于 `pwos-shared/link/pwos_link_frame.h`，是所有 STM32 有线通信的基础帧格式：

![Link Frame v2 帧结构与流式解析器](FinalPre/link_frame.svg)

```text
Offset  Size  Field
0       2     magic      = 'M' 'H'
2       1     version    = 2
3       1     hdr_len    = 19
4       1     type       (LINK/CTRL/DATA 类型)
5       1     flags
6       1     src         (源地址)
7       1     dst         (目标地址)
8       1     ttl         (存活跳数)
9       2     seq         (序列号)
11      2     ack         (确认号)
13      2     payload_len (≤ 512)
15      2     header_crc  (CRC-16/CCITT-FALSE，覆盖 magic..payload_len)
17      2     payload_crc (CRC-16/CCITT-FALSE，仅覆盖 payload)
19      N     payload
```

**关键特性**：

- 头 19 字节，payload 最大 512 字节，完整帧最大 531 字节
- 双 CRC 保护（头 CRC + payload CRC）
- 流式 parser 支持 DMA 分块输入、半帧保留、噪声重同步
- 中继转发时递减 TTL，TTL=0 或无路由时丢弃

![流式解析器有限状态机](FinalPre/parser_fsm.svg)

**帧类型表**：

| 值 | 名称 | 作用域 |
|---:|------|--------|
| `0x01` | `LINK_HELLO` | 物理邻居发现 |
| `0x02` | `LINK_HELLO_ACK` | HELLO 确认 |
| `0x10` | `CTRL_NODE_REGISTER` | 节点注册 |
| `0x11` | `CTRL_ADDR_ASSIGN` | 地址分配 |
| `0x12` | `CTRL_LEASE_RENEW` | 租约续期 |
| `0x13` | `CTRL_LEASE_ACK` | 续期确认 |
| `0x14` | `CTRL_LINK_STATE` | 链路状态上报 |
| `0x15` | `CTRL_ROUTE_UPDATE` | 路由下发 |
| `0x16` | `CTRL_HOST_ADVERTISE` | 主机 authority 通告 |
| `0x80` | `DATA_MINI9P` | mini9P 文件协议 |
| `0x81` | `DATA_RPC` | MCU RPC |
| `0x82` | `DATA_JOB` | Job System |
| `0x83` | `DATA_BULK` | 大数据（预留） |

#### 2.5.2 Mesh2 控制面协议

定义于 `pwos-shared/mesh2/pwos_mesh2_control.h`。控制 payload 版本为 1，使用固定长度
小端序编码：

| 消息 | 长度（字节） | 关键字段 |
|------|-------------|----------|
| `CTRL_NODE_REGISTER` | 24 | uid[3], boot_id, capability, upstream |
| `CTRL_ADDR_ASSIGN` | 28 | assigned_addr, lease_epoch, lease_ttl |
| `CTRL_LEASE_RENEW` | 24 | addr, lease_epoch |
| `CTRL_LEASE_ACK` | 28 | addr, new_lease_epoch, new_lease_ttl |
| `CTRL_LINK_STATE` | 24 | port_id, peer_uid, link_quality |
| `CTRL_ROUTE_UPDATE` | 12 | dst, next_hop, port, metric, version |
| `CTRL_HOST_ADVERTISE` | 28 | host_uid, epoch, priority, entry_port |

#### 2.5.3 mini9P 文件协议

![mini9P health读取时序](FinalPre/request_sequence.svg)

定义于 `pwos-shared/mini9p/mini9p_protocol.h`，保留了 Plan 9 的 9P 协议核心语义：

**消息类型**：

- `Tattach` / `Rattach`：建立连接，协商 msize
- `Twalk` / `Rwalk`：路径遍历，返回 Qid 序列
- `Topen` / `Ropen`：打开文件，返回 iounit
- `Tread` / `Treaddir` / `Rread`：读取文件内容或目录项
- `Twrite` / `Rwrite`：写入文件
- `Tstat` / `Rstat`：获取文件元信息
- `Tclunk` / `Rclunk`：释放 fid

**设计要点**：

- mini9P 完整帧作为 `DATA_MINI9P` payload 传输，中继不解析内容
- 主机路径 `/mcu2/sys/health` 在发往节点前由 `cluster_vfs` 转换为 `/sys/health`
- 通过全局 wire tag 和 `(DATA_MINI9P, src_addr, tag)` 匹配响应
- 固定大小 fid 表、静态缓冲区，避免动态内存管理

#### 2.5.4 MCU RPC 协议

![Typed Data Planes](FinalPre/dataplane.svg)

定义于 `pwos-shared/rpc/pwos_rpc_protocol.h`：

- **固定头**：16 字节（version, kind, flags, call_id, status, deadline, service_len,
  method_len, payload_len）
- **Kind**：request(0)、response(1)、cancel(2)、stream_chunk(3)、stream_end(4)
- **Flags**：oneway(0x01)、stream(0x02)
- **Streaming**：chunk 固定上限 32 字节，间隔 10ms；`status` 字段承载从 0 开始的
  chunk 序号；主机检测丢包、重复或乱序后终止 pending
- **Deadline**：毫秒级请求预算；超时后主机自动发送 CANCEL

**内置服务**：`system.ping`、`system.stream`、`system.info`、`system.notify`、
`system.delay`、`system.fail`

#### 2.5.5 Job 协议

![异步Job生命周期](FinalPre/job_sequence.svg)

定义于 `pwos-shared/job/pwos_job_protocol.h`：

- **固定头**：20 字节（version, kind, request_id, job_id, kernel, state, status, ...）
- **Kind**：caps、submit、status、result、cancel
- **State**：created → queued → assigned → running → done/failed/cancelled/lost
- **Kernel**：hash(FNV-1a)、vector_add、matmul(2×2)、mandelbrot、raytrace_tile(smallpt)
- **Request ID** 参与 typed pending；`job_id` 标识远端任务实例
- **LOST 机制**：节点重启、事务超时或远端 job 消失时标记 LOST；retry 创建新 job

### 2.6 分布式计算系统

#### 2.6.1 Job System 架构

Job System 是 9P-Walkers 中异步计算任务的统一管理和执行框架：

![增量计算Worker](FinalPre/compute_worker.svg)

```text
┌── 主机侧 (P4/S3) ──┐          ┌── 从机侧 (STM32) ──┐
│                      │          │                      │
│  job_manager         │  DATA_JOB│  job_service         │
│  ├─ 16 静态槽        │ ◄══════► │  ├─ 协议解析         │
│  ├─ LOST 检测        │          │  ├─ 4 静态 job 槽    │
│  ├─ retry 管理       │          │  └─ compute_worker 分发│
│  └─ deadline 超时    │          │                      │
│                      │          │  compute_worker      │
│  host_shell          │          │  ├─ 低优先级 task     │
│  └─ job 命令回调     │          │  ├─ 增量 step 推进    │
│                      │          │  └─ 320B 输入/输出缓冲│
└──────────────────────┘          └──────────────────────┘
```

**设计原则**：

- 主机侧 16 个静态 job 槽，保留原始输入以支持 LOST 后显式 retry
- 从机侧 4 个静态 job 槽，320 字节输入缓冲和 320 字节结果缓冲
- Compute worker 以低优先级 FreeRTOS task 运行，每次 step 只推进一个元素（一个字节、
  一个向量分量、一个矩阵元素或一个像素），保证控制面实时性
- Service task 只做协议解析和 job 状态转换，不等待计算完成

#### 2.6.2 支持的 Kernel

| Kernel | 输入 | 输出 | 用途 |
|--------|------|------|------|
| `hash` (FNV-1a) | 任意字符串 | 32-bit hash | 功能验证 |
| `vector_add` | 向量长度 N | N 维向量 | 功能验证 |
| `matmul` | 2×2 矩阵 | 2×2 矩阵 | 功能验证 |
| `mandelbrot` | 宽度×高度×迭代次数 | 字符画 | 功能验证 |
| `raytrace_tile` (smallpt) | 22 字节 tile 请求 | RGB565 像素数组 | 分布式渲染 |

#### 2.6.3 光线追踪渲染管线

基于 smallpt（Small Path Tracer）的全局光照渲染器，场景包含：

![Cornell Box 光线追踪场景](FinalPre/raytrace_classic.svg)

- **Cornell Box** 五面 diffuse 墙壁（红/绿/白）
- **左侧 mirror 球**（完全镜面反射）
- **右侧 glass 球**（折射 + 菲涅尔反射）
- **顶部面光源**（重要性采样直接光照）

**渲染参数**：4 samples/pixel，4 次弹射深度，Whitted 风格递归路径追踪。

**分片调度流程**（P4 Lua 调度器 `whitted_scheduler.lua`）：

1. 动态枚举在线 MCU 列表
2. 将 240×320 场景按 16×7 tile 拆分（共 16×46 = 736 个 tile）
3. 为每个 tile 编码 22 字节请求：`(protocol_v3, scene_id, tile_x, tile_y, tile_w,
   tile_h, image_w, image_h, samples, max_depth, frame_id, seed, camera_phase)`
4. 每个 MCU 一个活跃 tile（最多分配一个，完成后分配下一个）
5. 计算结果为 RGB565 像素数组（16×7×2 = 224 字节）
6. 结果写入 F429 的 `/display/tile`，由 LCD 控制器显示
7. 支持多帧累积和摄像机相位旋转动画

**显示**：STM32F429 通过 LTDC 驱动 ILI9341 LCD，240×320 RGB565 持续刷新。

### 2.7 LLM 推理引擎

#### 2.7.1 引擎概述

基于 Andrej Karpathy 的 [llama2.c](https://github.com/karpathy/llama2.c) 移植，
适配 ESP-IDF (FreeRTOS + PSRAM + 双核并行)：

- **模型**：stories260K（轻量级故事生成模型）
- **架构**：Transformer decoder-only，支持 GQA（Grouped Query Attention）
- **双核并行**：Core-0 运行 helper task（matmul/forward 子任务），Core-1 运行主循环
- **内存布局**：模型权重（~1.1 MB）和 KV Cache（~0.64 MB）分配在 PSRAM

#### 2.7.2 内存预算（stories260K @ ESP32-S3）

| 组件 | 大小 | 位置 |
|------|------|------|
| 模型权重 (checkpoint) | ~1.10 MB | PSRAM |
| KV Cache (key + value) | ~0.64 MB | PSRAM |
| RunState 其他缓冲区 | ~0.03 MB | PSRAM |
| Tokenizer 词表 | ~0.02 MB | PSRAM |
| 输出缓冲区 | ~0.004 MB | PSRAM |
| **合计** | **~1.80 MB** | |
| ESP32-S3 PSRAM 总量 | 2 MB (最小) / 8 MB | |
| 可用余量 (扣除 WiFi/LWIP) | ~0.1 MB (2MB) / ~6 MB (8MB) | ⚠️ 紧张 |

**优化建议**：优先使用 8 MB PSRAM 模组，或将 `seq_len` 从 256 降至 128（KV cache 减半）。

#### 2.7.3 推理运行时架构

```text
app_main (Core-1)
  --> pwos_inference_runtime_start()
       --> inference_worker task  (Core-1, 优先)
            --> llm_engine.generate()
                 --> forward() x N tokens
                      --> matmul_task  (Core-0)  双核并行矩阵乘法
                      --> forward_task (Core-0)  双核并行 attention/FFN
       --> inference_console task (Core-1, 低优先)
            --> WebSocket → 用户

dist_inference_service (Core-1)
  --> host RPC 接口
  --> 跨主机推理请求转发
```

**核心安全机制**（经过全面审计后的改进）：

1. **EOS 回退保护**：`pos=0` 时输出 EOS 不回退到负值
2. **大块分配不回退**：>32 KB 分配失败不尝试内部 SRAM（避免堆破坏）
3. **超时机制**：所有同步原语（semaphore/event group）设有超时（2s/5s）
4. **栈空间充足**：matmul_task/forward_task 栈 6144B，app_main 栈 8192B
5. **TWDT 集成**：Task Watchdog Timer 监控 idle tasks，防止死锁
6. **启动阶段追踪**：`boot_phase_t` 枚举 + 30 秒超时重启
7. **PSRAM 自检**：每 30 秒检查 PSRAM 空闲量，< 128 KB 主动重启

### 2.8 渲染与显示系统

#### 2.8.1 P4 Lua 渲染调度器

位于 `pwos-master-esp32p4/render/`，使用 Lua 5.4 虚拟机执行调度脚本：

- **动态节点发现**：通过 `cluster.nodes()` 获取在线 MCU 列表
- **显示节点探测**：`display.probe(node)` 查找 F429 显示节点
- **Tile 构建**：按 16×7 规格将 240×320 画布拆分为 tile 队列
- **并发调度**：每个 MCU 同时最多一个活跃 tile，完成后立即分配下一个
- **结果转发**：`display.blit(display_node, pixels)` 将 tile 写入 F429 LCD
- **多帧动画**：每帧旋转摄像机相位（+2°），生成连续旋转动画

#### 2.8.2 smallpt 光线追踪 Kernel

位于 `pwos-shared/render/pwos_smallpt.c`：

- 协议版本 v3：tile_x/tile_y 升级为 16-bit（支持 320+ 行分辨率）
- 请求 22 字节，结果最大 240 字节（headers 16B + 16×7×2B pixels）
- 支持 Whitted 场景（镜面反射 + 折射 + 漫反射 + 面光源直接光照）
- 使用 xorshift32 伪随机数生成器
- 所有向量运算内联（避免函数调用开销）

#### 2.8.3 F429 显示后端

STM32F429 显示子系统：

- **硬件**：LTDC 控制器 + SDRAM 帧缓冲 + ILI9341 LCD（240×320）
- **命名空间**：`/display/status`（只读）、`/display/tile`（只写）
- **数据格式**：RGB565 little-endian（16-bit/像素）
- **写入方式**：通过 mini9P Twrite 或 Job result 自动转发

仅在 Debug 构建中启用，支持以下注入类型：

| 命令 | 说明 |
|------|------|
| `fault <mcu> drop <port> <n>` | 丢弃端口 n 个帧 |
| `fault <mcu> delay <port> <ms>` | 端口延迟 ms 毫秒 |

**光线追踪整体性能**：

- 240×320 场景 = 736 个 tile（16×7 每 tile）
- 3 个 STM32 并行：736 ÷ 3 ≈ 245 tile/node
- 单帧预计耗时：245 × 30s ≈ 2 小时（4 samples/pixel）
- 实际演示中使用较少 samples 和 depth 以减少帧时间

**注意**：STM32F407 运行在 168MHz，无 FPU 硬件加速的浮点运算（光线追踪中大量使用），
因此 raytrace tile 计算时间较长。这体现了 MCU 集群计算的典型特征：单节点算力有限，
但可通过并行扩展提升总吞吐。

## 4. 总结与展望

### 4.1 项目成果

9P-Walkers 项目在课程周期内成功实现了一个功能完备的 MCU 集群分布式实时操作系统原型，
核心成果包括：

1. **完整的协议栈**：从 link frame v2（数据链路层）到 mesh2（控制面）、mini9P（文件协议）、
   MCU RPC（远程调用）、Job（异步计算）、Host RPC（主机间通信），形成了五层协议体系

2. **统一命名空间**：通过 `cluster_vfs` 和 mini9P 协议，将多个异构 MCU 的文件树聚合到
   `/mcuN/...` 路径下，实现了 Plan 9 风格的"一切皆文件"抽象

3. **分布式计算**：Job System 支持异步任务提交、状态查询、结果获取、取消和故障恢复；
   基于 smallpt 的光线追踪渲染器可在多个 STM32 上分片并行计算，结果实时显示在 F429 LCD

4. **本地 LLM 推理**：成功将 llama2.c 移植到 ESP32-S3，实现了 Transformer 模型在 MCU
   上的本地推理，并经过全面安全审计保证了系统稳定性

5. **多主机协同**：ESP32-P4 (Ethernet) 和 ESP32-S3 (WiFi) 通过 mDNS 发现、epoch/priority
   选主、CBOR host RPC 实现跨主机节点访问

6. **完善的测试体系**：14 组独立 PC 端 CTest、10 个里程碑验收、系统级故障注入和长时间
   稳定性测试

7. **可观测性**：完整的主从 `/sys` 诊断命名空间、WebShell 远程交互界面、15 条系统命令

### 4.2 技术亮点

- **ISR 最小化原则**：中断服务例程只搬运数据和通知任务，所有协议逻辑在任务上下文中执行
- **Typed Pending 机制**：通过 `(data_type, src_addr, tag)` 三元组精确匹配并发响应
- **静态内存管理**：所有热路径使用固定上限的静态内存，零动态分配
- **分层状态所有权**：link、端口、控制面、service、compute 各模块严格拥有各自状态
- **Boot ID 隔离**：节点重启后旧 session、fid、pending 全部失效，避免旧响应污染新实例
- **LOST 恢复**：节点故障后 job 标记 LOST，恢复后通过 retry 创建新 job 而非复活旧 ID
